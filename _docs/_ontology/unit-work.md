# 단위 작업 (unit work)

## 정의

**의미적으로 atomic한 작업 단위.** 하나의 [작업 트리](../_worklog/TASK_TREE.md) 노드로 분류 가능한 의미를 가진 일감.

핵심은 commit·브랜치 단위가 아니라 **"구분할 수 있는 의미의 단위"**다. 직후에 바로 작업되는 정정 commit은 별도 단위 작업이 아니라 **앞 작업과 같은 단위 작업의 일부**로 간주한다 — 의미는 정정 전후가 같은 한 묶음이기 때문.

크기 기준의 직관: 보통 풀 리퀘스트(pull request) 1개가 자연스러운 일감 크기다. 그 정도가 단위 작업의 표준 크기.

## 재귀 쪼개짐

작업 트리는 깊이를 가진다. 단위 작업도 그에 따라 재귀적으로 쪼개진다.

- 1뎁스에서 보면 큰 단위 작업 (예: "사용자 공유 영상 3건 ingest 라운드")
- 그 자식 노드가 더 세부적 단위 작업 (예: 영상 1건 ingest)

어느 깊이를 단위 작업으로 볼지는 작업 트리의 어느 노드에 풀 리퀘스트(또는 메인 머지)를 묶는가의 문제다.

## 작업 트리·브랜치와의 대응

- 단위 작업 ↔ 작업 트리 한 노드 (의미적 1:1)
- 단위 작업 ↔ 브랜치는 운영 모델에 따라 갈린다.
  - **TBD(트렁크 기반 개발)**: 단위 작업은 보통 1뎁스 깊이로 잡혀 메인 머지 1회 = 브랜치 1개에 대응. 깊은 작업 트리의 하위 노드는 같은 브랜치 안 commit으로 처리되는 경향.
  - **Git Flow**: 피처 브랜치 통짜가 단위 작업일 수도, 그 안 하위 노드(서브 브랜치·풀 리퀘스트)가 단위 작업일 수도. 깊이마다 풀 리퀘스트가 달릴 수 있어 더 유연.

## 운영 권장 (TBD 환경)

TBD 환경에서는 단위 작업과 작업 트리 노드의 대응이 규격상 명확하지 않다. 어느 노드에 대한 브랜치인지가 브랜치명만으로는 결정 안 됨. 그래서 다음 순서가 권장:

1. **작업 트리 점검·정리**: 새 작업 시작 전 [TASK_TREE.md](../_worklog/TASK_TREE.md)를 보고 어느 노드에 대한 작업인지 정함. 노드가 없으면 신설.
2. **단위 작업 깊이 결정**: 그 노드를 통째로 한 단위 작업(=한 브랜치)으로 잡을지, 자식들을 별도 단위 작업으로 쪼갤지 결정.
3. **브랜치 생성**: 결정된 단위 작업 슬러그로 브랜치 따기.
4. **commit**: 단위 작업의 자식들·정정·미세 보강을 같은 브랜치 안 commit들로 누적.

작업 트리 정리 없이 브랜치를 먼저 따면 단위 작업의 의미 단위가 흐려진다.
