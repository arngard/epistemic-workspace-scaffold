# 온톨로지 디스크립터

> "우리 도메인에 어떤 것들이 존재하고, 어떤 관계를 맺는가" - 도메인 개념 정의.
> 같은 단어를 두 사람(또는 두 문서)이 다른 뜻으로 쓸 수 있는 경우에만 정의한다. 보편적인 용어는 정의하지 않는다.
> 참조 폴더이므로 자기설명적 파일명이 인덱스를 대신한다. "어떤 문서가 존재하는가"의 SSOT는 파일시스템이다 (cf. [AGENTS.md](../../AGENTS.md) "폴더 디스크립터 사용 규칙").

## 고유 규칙

- 코드에서 도메인 클래스를 설계하기 전에 이 폴더에서 해당 개념의 정의를 확인한다.
- 정의를 바꾸면 관련 `_architecture/` 및 `_strategy/` 문서와의 정합성을 확인한다.

## 작성 스타일

- 정의를 먼저 서술하고, 예시나 맥락은 그 뒤에 둔다.
- 보편적이고 일반적인 용어는 정의하지 않는다. 이 도메인에서만 구분이 갈리는 용어에 한정한다.
- 관련 개념은 마크다운 링크로 연결한다.
- 개별 용어 정의는 반 페이지 이내가 권장이며, 더 길어지면 개념을 쪼갤 신호로 본다.

## 용어 도입 전 검증

새 용어를 온톨로지에 편입하거나 다른 문서에서 채택하기 전 다음을 확인한다. 이 검증을 건너뛰면 이후 이름 사후 일괄 교체 비용이 크다 (cf. [AGENTS/conventions.md](../../AGENTS/conventions.md) "이름과 식별자의 안정성").

- 타사 제품, 툴의 고유명사인가. 그렇다면 우리 용어로 내재화하지 않는다. 외부 대면 명칭은 별도 결정 대상.
- 일반어와 의미가 어긋나지 않는가. "시스템 이벤트"가 OS 이벤트로 해석되는 식의 혼동이 없는지.
- 단수/복수, 단일/다중 구조를 잘못 암시하지 않는가. 실제로는 여럿/복합인 것을 단일, 만능 요소로 오해하게 만드는 이름이 아닌지.
- 레이어, 패키지, 모듈명으로도 쓰일 단어인가. 역할명과 레이어명이 겹치면 혼동이 발생한다.

## 스캐폴드 샘플

`sample-term.md`는 용어 정의 문서의 위치와 형식을 보여주는 튜토리얼용 샘플이다. 실제 작성 시 삭제하거나 교체한다. `unit-task.md`는 본 스캐폴드가 계속 참조하는 실제 정의(단위 작업)이므로 유지한다.
