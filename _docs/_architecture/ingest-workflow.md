# 새 정보 ingest 워크플로

워크스페이스에 새 사실·의견·예측·결정·자료가 들어올 때 AI가 따르는 표준 절차. 빠뜨림과 정합성 부채를 줄이는 것이 목적이다.

## 적용 범위

다음 입력에 적용한다.

- 사용자가 대화 중에 알린 새 사실·의견·예측·결정.
- `_reference/` 폴더에 사용자가 dump한 자료 파일.
- 외부 도구(웹 검색, fetch, MCP 등)에서 가져온 사실.

도메인 작업 진행 중 자체 추론으로 도달한 결론은 별개다. 그건 `_strategy/`나 분석 문서로 들어간다.

## 표준 시퀀스

각 단계를 건너뛰지 않는다. 체감상 과한 단계가 있어도 누적 정합성 부채를 막는 비용으로 본다.

1. **분류**: 들어온 입력이 사실(외부 관찰)인가, 의견(주관적 진술)인가, 예측(미래 추정)인가, 결정(우리 측 판단)인가, raw 자료인가.
2. **출처·시점·주체 식별**: 사실은 출처와 발견 시점, 의견·예측은 화자와 발언 시점·강도, 결정은 결정 주체와 시점. 이 메타가 빠지면 이후 가중치 재구성이 불가능하다.
3. **기존 노드 검색**: 같은 주제·관련 의견·동일 출처·동일 화자·동일 결정 주제의 기존 노드를 찾는다.
   - 우선 `_docs/_*/INDEX.md`(필요하면 sub-INDEX)를 훑어 카테고리 후보 파악.
   - 후보 노드 본문은 Grep으로 핵심 키워드 검색해 위치 확인.
4. **충돌·중복 검사** (Contradiction flagging):
   - 새 입력이 기존 노드의 진술과 충돌하면 조용히 덮어쓰지 않는다.
   - 두 진술을 모두 보존하고 차이를 명시한다 (예: "X 화자는 시점 T1에 ...라고 했지만 시점 T2에 ...로 갱신", "A는 ...라고 주장, B는 반대 주장").
   - 동일 화자의 시간차 의견 변화는 그 화자 또는 의견 노드에 시점 함께 누적.
   - 충돌 의미가 모호하면 사용자에게 짚어 묻는다 — AI가 단정하지 않는다.
5. **카테고리 결정**: [AGENTS/how-to-separate-docs-folders.md](../../AGENTS/how-to-separate-docs-folders.md) 판별 규칙 적용.
6. **노드 결정**: 기존 노드에 추가 vs 신규 노드 생성. 기존 노드의 주제 응집성을 깨면 신규로 분리 (한 문서는 한 관심사 원칙).
7. **교차 참조 갱신**: 같은 정보가 여러 노드에 걸쳐 있으면 노드 간 마크다운 링크로 연결한다. 워크스페이스가 양방향 그래프 패턴(예: 화자↔의견)을 채택했다면 그 양방향 의무도 같은 단계에서 처리한다.
8. **INDEX 갱신**: 신규 노드면 해당 INDEX 또는 sub-INDEX에 등록. 기존 노드 갱신이면 INDEX의 설명·`date modified`·등록 정보를 점검.
9. **종료 전 빠뜨림 검사**: 위 단계 중 누락이 있는지 자체 review.
   - 출처·시점·주체를 빠뜨렸는가?
   - 기존 노드 검색을 했는가? 충돌 검사를 했는가?
   - INDEX 등록·갱신을 했는가?
   - 양방향 링크를 한쪽만 갱신하지 않았는가?
   - 새 사실·결정이 관련 strategy 또는 architecture 문서를 무효화하지는 않는가?

## `_reference/` 폴더 처리

`_reference/`는 사용자가 임의로 dump하는 자료의 immutable 보관 영역이다. 형태 무관 (PDF, 이미지, 텍스트 메모, URL 모음 등). cf. [`_reference/INDEX.md`](../../_reference/INDEX.md).

### 처리 여부 식별

[`_reference/INDEX.md`](../../_reference/INDEX.md)가 인벤토리이자 처리 진척의 SSOT다. 별도 처리 로그 파일은 두지 않는다.

세션 시작 시 또는 작업 중 트리거가 있을 때:

1. `_reference/` 내용 listing.
2. 본 INDEX와 대조.
3. 폴더에는 있지만 INDEX에 없으면 신규 미처리 자료. INDEX에 등재하고 처리 절차 시작.
4. INDEX에 있으나 폴더에는 없으면 raw 자료가 사라진 것. 사용자에게 짚는다.
5. INDEX 상태 컬럼이 `처리 대기`인 항목은 처리 절차 진행 또는 사용자에게 우선순위 확인.

보조 검증으로 본문에서 `_reference/<file>` 경로를 grep해 derived 노드의 존재를 교차 확인할 수 있다 (INDEX 상태와 실제 cross-reference의 일치 점검).

### 미처리 항목 처리

미처리 항목 발견 시:

1. 자료 내용을 읽고 어떤 카테고리·노드에 통합할지 판단.
2. 명확하면 위 표준 시퀀스로 통합 후 derived 노드에 raw 파일 경로를 reference로 명시 (`cf. _reference/<file>`). INDEX 상태를 `처리됨`으로 갱신하고 derived 노드 링크를 함께 적는다.
3. 모호하거나 사용자 의도가 불분명하면 INDEX 상태를 `사용자 의견 필요`로 두고 사용자에게 의견을 묻는다.

### Raw 자료 자체는 immutable

처리 후에도 `_reference/` 안의 파일은 삭제·수정하지 않는다. 원본 보존이 SSOT다. derived 노드는 변경되더라도 raw 입력으로 항상 되돌아갈 수 있어야 한다.

## 빠뜨림 방지의 명시적 의도

본 절차의 표준 시퀀스 9번 "종료 전 빠뜨림 검사"는 의도적인 자체 점검 단계다. 워크스페이스 작업 중 화자 등록·INDEX 갱신·양방향 링크 한쪽 등 자잘한 작업이 누락되는 사례가 관찰됐고, 이 절차는 그 누락을 줄이는 것을 명시 목적으로 한다.
