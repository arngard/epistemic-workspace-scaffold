# 핵심 제약

프로젝트 전반에 걸쳐 적용되는 설계 제약. 개별 상세는 각 링크 문서를 참조한다.

## 0. 설계 원칙 (tiebreaker)

설계 판단이 갈릴 때의 우선 기준:

- 대칭성(symmetry): 같은 역할의 요소는 같은 형식을 갖는다. 레이어 간·플랫폼 간 동일 패턴.
- 일관성(consistency): 예외적 패턴을 최소화한다. 한 번 익힌 패턴이 어디서나 재사용된다.
- 예측 가능성(predictability): 새 요소를 추가할 때 그 위치와 형식이 기존 패턴에서 자동으로 도출된다.
- 강타입(strong typing): 데이터 모델을 엄밀하게 정의하여 컴파일 타임에 오류를 잡는다. `Any`, 타입 소거 래퍼, 느슨한 딕셔너리를 지양하고, 모든 값에 구체적 타입을 부여한다. 구조적으로 올바르게 만들어서 컴파일되면 맞게 돌아가는 구도를 추구한다 (correctness by construction). 테스트(귀납적 검증)는 이 위에 보완적으로 얹는 것이지, 타입이 잡아야 할 오류를 테스트에 위임하지 않는다.

이 원칙은 모든 규약과 구조 결정의 최후 tiebreaker이다.

## 원본 보호 원칙

에이전트가 원본(데이터, 콘텐츠)을 다룰 때의 안전 기준. 가역성이 1순위, 공유 기준선 보호가 2순위다. 둘은 독립적으로 작동하며, 변경이 둘 중 하나라도 게이트에 걸리면 사용자 확인을 거친다.

1. 가역성 우선 (1순위): 되돌릴 수 없는 손실은 공유 여부와 무관하게 가장 먼저 막는다. 소스 관리(원격 저장소, 커밋 이력)로 원본을 복원할 수 있는 영역은 에이전트가 직접 작업한다. 복원 수단이 없어 로컬의 그 영역 자체가 유일한 원본일 때만, 직접 변경을 피하고 사용자 확인을 거친다. 판단 축은 특정 행위(머지 같은)가 아니라 복원 가능성이다. 비가역 손실은 복구 비용이 사실상 무한대이므로 최우선으로 둔다.

2. 공유 기준선 보호 (2순위): 가역성이 확보된(복원 가능한) 범위 안에서 추가로 적용된다. 여러 주체(다른 세션, 협업자, 자동 트리거)가 검토된 상태로 신뢰하는 공유 기준선(`main` 브랜치, `_docs/` SSOT 등)에 변경을 반영할 때는 사용자 검토를 거친다. 복원 가능성과 별개로, 검토되지 않은 상태가 기준선이 되는 것을 막는 신뢰 게이트다. `main` 직접 커밋 금지, 사용자 검토 통과 후 머지(자율 머지 회피), `_docs/` 사람 확인 없는 대폭 재편 금지가 모두 이 원리의 사례다(AGENTS.md "경계" 절).

판단 순서: 먼저 복원 가능성을 본다. 복원 불가능하면 공유 여부와 무관하게 사용자 게이트(가역성). 복원 가능하면 그 다음 공유 기준선인지로 판단한다(공유면 검토 게이트, 비공유 작업 영역이면 자율). 두 고려가 갈리는 모호한 경계에서는 가역성을 따른다.
