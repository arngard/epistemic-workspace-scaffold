# 전략 색인

> "우리가 왜 이렇게 하기로 했는가" — 우리의 결정과 그 근거.
> 외부 사실과 환경 정보는 [_knowledge/](../_knowledge/INDEX.md) 참조.
> 최종 설계 명세는 [_architecture/](../_architecture/INDEX.md) 참조.
> 공통 사용 규칙은 [AGENTS.md](../../AGENTS.md)의 "색인(INDEX) 사용 규칙" 섹션 참조.

## 작성 스타일

전략 문서는 아래 형식을 따른다.

### Front matter 속성

일반 필수 필드 외에 다음 속성을 추가한다.

```yaml
importance: 상 | 중 | 하
urgency: 상 | 중 | 하
```

속성은 각각 독립적으로 판단한다. 값 선택의 근거는 별도 필드로 두지 않고 "상황" 섹션 서술에 자연스럽게 녹인다.

#### importance

- 상: 거시적 장기 전략, 방향성.
- 중: 평범한 실행 전략, 실행안에 대한 결정.
- 하: 자잘한 추구, 권고.

#### urgency

- 상: 다른 것을 제쳐두고 먼저 실행해야 함. 모든 상급은 중급보다 우선한다.
- 중: 일반적인 우선순위 스케줄링 대상.
- 하: 밀려도 무방.

단, 의존성이 urgency보다 우선적인 판단 기준이다. 의존성 때문에 urgency 상급 항목이 밀릴 수 있다. 이 경우 작업 항목을 재분할하거나 우선순위를 재할당하는 것을 제안한다.

### 섹션 구조

1. 상황: 배경과 맥락. 피상적으로 드러나는 문제 인식도 여기에 함께 서술한다.
2. 난점: 대응안 고안을 막고 있는 장애물·선결 과제. 대응이 확정되면 이 섹션은 삭제한다 — 섹션이 살아 있다는 사실 자체가 "결정 미완료" 상태를 의미한다.
3. 대응: 결정 사항. 기대 효과는 "~하면 ~할 수 있다" 형식으로 서술에 녹인다. 대응 요소와 결과 요소가 1:1 대응이 아니므로 "기대 효과"를 별도 섹션으로 분리하지 않는다.
4. 참고: 링크·레퍼런스.

### 여러 하위 결정을 포함하는 전략 문서

한 전략 문서가 여러 하위 결정(예: 한 모듈의 구체 스펙 + 에러 타입 + 정책)을 담을 때:

- 확정된 하위 결정은 "대응" 또는 본문 하위 섹션에 선언문으로 둔다.
- 미결인 하위 결정만 "난점" 섹션에 남긴다. 모두 확정되면 "난점"을 통째로 삭제한다.
- "결정이 필요한 것들"처럼 확정 상태가 드러나지 않는 오픈 리스트 형식은 쓰지 않는다. 독자가 무엇이 확정이고 무엇이 미결인지 판단할 수 없다.
- 내부 재편만으로 감당이 안 될 만큼 커지면 새로운 시각에서 문서 자체의 분할·재배치를 검토한다. 별도 전략 문서로 쪼개거나, 내용의 성격에 따라 재배치할 수 있다. 이 경우 일반적인 기준에 따라 재검토한다. cf. [how-to-separate-docs-folders.md](../../AGENTS/how-to-separate-docs-folders.md).
