# 워크스페이스 구조

## 목표

하나의 레포를 워크스페이스로 다루면서, 그 안에 설계 문서와 프로젝트를 모두 포함한다.

## 네이밍 규칙: 워크스페이스 scaffolding과 콘텐츠

폴더는 두 종류로 나뉜다.

- 워크스페이스 scaffolding 폴더 (`_` 접두어): 워크스페이스를 처음 세팅할 때 만들어지는 고정 골격. 정보가 어디에 놓이는지를 결정하는 구조. 프로젝트 수명 동안 추가되거나 사라지지 않는다.
- 콘텐츠: 작업하면서 생기고 변하는 실제 산출물. 문서 파일, 프로젝트 폴더 등. `_` 접두어를 붙이지 않는다.

`_` 접두어는 파일시스템 정렬에서 워크스페이스 scaffolding 폴더를 앞쪽에 모아주는 부수 효과도 있다.

### 하위 문서 폴더 규칙

어떤 문서가 길어져서 여러 하위 문서로 쪼개야 할 때, 상위 문서와 같은 이름의 폴더를 만들어 하위 문서를 넣는다. 예: `AGENTS.md`의 하위 문서들은 `AGENTS/`에 둔다.

## 최상위 구조

```
(workspace-root)/                      ← 워크스페이스 루트
├── AGENTS.md                           ← AI 에이전트 공통 지침 (모든 도구의 진입점)
├── AGENTS/                             ← AGENTS.md의 하위 문서
├── _docs/                              ← 코드로 표현할 수 없는 지식
│   ├── _worklog/                       ← 작업 상태, 이력, 작업 트리
│   ├── _ontology/                      ← 도메인 개념 정의 ("무엇이 존재하는가")
│   ├── _knowledge/                     ← 외부 사실, 환경, 벤치마크 ("세상이 어떠한가")
│   ├── _strategy/                      ← 우리의 결정과 근거 ("왜 이렇게 하기로 했는가")
│   └── _architecture/                  ← 설계 명세 ("어떻게 만드는가")
└── _implementation/                    ← 프로젝트
```

`_docs/` 하위 폴더에 새 문서를 배치할 때의 분류 기준은 [how-to-separate-docs-folders.md](how-to-separate-docs-folders.md) 참조.

## 문서와 코드의 역할 분담

`_docs/`는 코드로 표현할 수 없는 것 위주로 담는다. 도메인 개념, 외부 사실, 결정 근거, 상위 수준의 구조 결정이 이에 해당한다.

구현이 시작되면, 모듈의 상세 스펙(인터페이스 시그니처, 데이터 타입, 에러 계약)은 코드 자체가 명세가 된다. 확정된 추상화 레이어에는 `[to:@AGENT: 확정된 추상화 레이어]` 형식의 공용 태그 주석을 달아 변경을 통제한다. 태그 형식은 [agent-tag-format.md](agent-tag-format.md) 참조.

코드와 1:1 대응되는 대규모의 명세 문서를 별도로 유지관리하지 않는다. 문서에서 코드의 동작을 설명하거나 맥락을 제공하는 것은 자연스러운 일이며, 그것 자체가 문제는 아니다.

| 위치 | 담는 것 |
|------|---------|
| `_docs/` | 코드로 표현할 수 없는 것: 도메인 개념, 외부 사실, 결정 근거, 상위 구조 결정. |
| 코드 (인터페이스 + 공용 태그 주석) | 모듈 인터페이스, 타입, 에러 계약. 코드가 곧 명세. 확정된 레이어는 `[to:@AGENT]` 태그로 보호. |
| 코드 (구현) | 모듈 내부 구현. 자기설명적 코드 + 필요 시 주석. |
