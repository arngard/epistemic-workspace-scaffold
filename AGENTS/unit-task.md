# 단위 작업과 워크플로우

단위 작업이 무엇이고, 그 내부를 수행하는 워크플로우가 어떻게 조립되는가. 운영 절차는 [unit-task-workflow.md](unit-task-workflow.md)가 다룬다.

## 단위 작업

의미적으로 원자적인(atomic) 작업 단위. 트랜잭션처럼 하나로 완결된다. 크기는 정해져 있지 않다 - 의미로 나눌 수 있으면 나눈 것 하나가 단위 작업이다.

단위 작업은 [작업 트리](../_docs/_worklog/TASK_TREE.md) 노드 하나에 대응한다. 리프 노드만이 아니라 중위 노드도 하나의 단위 작업이며, 중위 노드 단위 작업은 자식 단위 작업들을 포괄한다 - 트랜잭션 안에 서브트랜잭션이 있는 것과 같은 중첩 구조다.

리프/중위는 노드의 종류가 아니라 위치 구분이다. 트리의 모든 작업 노드는 동일한 "작업"이고(cf. [TASK_TREE.md](../_docs/_worklog/TASK_TREE.md) 프리앰블 "깊이별 타입 구분 없음"), 리프냐 중위냐는 그 노드가 재귀할 자식을 갖느냐일 뿐이다. 워크플로우 형태가 갈리는 것도 이 차이에서 나온다 - 자식이 있으면 재귀 진행, 없으면 그 자리의 구체 작업.

- 커밋, 브랜치 단위가 아니라 의미 단위다. 직후에 바로 이어지는 정정 커밋은 별도 단위 작업이 아니라 앞 작업과 같은 단위 작업의 일부다 - 의미가 정정 전후로 같은 한 묶음이기 때문.
- 앞뒤 트리 갱신: 착수 직전에 해당 노드를 `[/]`로, 완료 직후에 `[x]`로 바꾼다. 이 두 갱신이 단위 작업의 시작과 끝을 표시한다.

## 워크플로우

워크플로우는 한 단위 작업의 내부를 어떻게 수행하는가의 절차다. 노드의 위치에 따라 형태가 다르며, 조립 가능한 모듈이다.

중위 노드의 워크플로우는 재귀적이다.

```
[진입] 내 노드 [/]
[진행]
  내가 할 일(있으면)
  최우선 자식 수행        <- 그 자식의 워크플로우를 다시 밟음 (재귀)
  내가 할 일(트리 갱신)    <- 자식 [x], 재평가
  다음 최우선 자식 수행    <- 재귀
  내가 할 일(트리 갱신)
  ...
[종료] 내 노드 [x]
```

자식 수행이 그 자식의 워크플로우를 다시 밟는 재귀 호출이므로, 앞뒤 트리 갱신은 모든 레벨에서 일어나고 중위 노드에서는 자식들 사이에도 일어난다.

리프 노드의 워크플로우는 자식 재귀가 없고 그 자리의 구체 작업 자체다. 이 작업이 특정 유형이면 그 유형의 워크플로우가 여기에 조립된다 - 새 정보 유입이면 [ingest-workflow.md](ingest-workflow.md), 변경을 검토받아 main에 머지하는 단계면 [pr-workflow.md](pr-workflow.md). 조립되는 워크플로우는 단독으로도 재사용된다.

즉 unit-task-workflow는 일반 프레임(진입/진행/종료 + 재귀)을 정의하고, 구체 워크플로우는 그 프레임의 정해진 단계에 조립된다. 이것이 "워크플로우는 조립된다"의 뜻이다.
