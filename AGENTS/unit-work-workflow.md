# 단위 작업 운영 흐름

## 단위 작업이란

한 호흡으로 묶이는 작업 단위. 단발 한 줄 수정·자료 1건 단편 ingest는 단위 작업이 아니라 자율 사이클의 자잘한 갱신에 가깝다. 다음 중 하나에 해당하면 단위 작업이다.

- 자료 N건 묶음 ingest 라운드
- 메타 결정·정책 변경 라운드 (예: 갱신 규칙 신설, 카테고리 재편)
- 큰 마이그레이션·구조 재편 (예: 폴더 분리, 파일 일괄 리네임)
- 도메인 모듈 1건 신설 (예: 새 sub-INDEX 도입과 첫 배치)
- 컨텍스트가 한 세션을 넘어갈 가능성이 큰 작업

모호 사례는 단위 작업 흐름이 안전. 끊겼을 때 회복 비용이 단발 처리보다 작다.

## 진입 흐름 (단위 작업 시작)

순서를 지킨다. 한 단계도 건너뛰지 않는다.

1. **작업 브랜치 생성** (cf. [branch-strategy.md](branch-strategy.md)): `<type>/YYMMDD-<짧은-설명>` 형식. 단위 작업마다 격리된 commit 라인을 확보. 라운드 도중 컨텍스트 압축·세션 끊김이 일어나도 작업 라인 추적이 깔끔하다.
2. **워크로그 등재**: `_docs/_worklog/STATUS.md`의 "현재 단계" 갱신 (작업 내용 + 시작 일시), `_docs/_worklog/TASK_TREE.md`에 라운드 루트 노드 + 처리 단위(자료 목록·하위 단계)를 자식 노드로 등재. STATUS·TASK_TREE는 다음 세션 진입자가 작업 컨텍스트를 복원하는 1차 입구다.
3. **시작점 commit**: 워크로그 등재만 담은 commit으로 작업 시작점을 git log에 마킹. `--allow-empty` 라도 만들어 두면 작업 브랜치 base와 첫 실작업 사이에 명확한 anchor가 생긴다. (워크로그 변경이 있으면 그걸 포함, 없으면 빈 commit.)

## 진행 흐름 (작업 중)

- 처리 단위(자료 1건·하위 단계 1개) 완료마다 또는 2-3건 단위로 commit. 컨텍스트 부담이 큰 작업은 더 작은 단위로 끊는다.
- TASK_TREE의 처리 단위 노드를 `[x]`로 갱신. 새로 드러난 항목은 즉시 추가.
- 컨텍스트 압축이 임박하면 STATUS의 "현재 단계"를 다음 세션이 이어받을 수 있는 형태로 다듬는다 (어디까지 진행, 다음에 무엇).
- 작업 브랜치는 자유롭게 push 해도 되고 안 해도 된다. 다만 세션이 끊길 가능성이 있으면 push 해 두는 게 안전.

## 종료 흐름 (단위 작업 끝)

순서를 지킨다.

1. **마무리 commit**: 작업 마지막 정리(미캡처 보강·표준 패턴 추가 학습 사항 반영·STATUS 마지막 작업 일시 갱신 등)를 담은 commit. 이 시점까지의 모든 작업은 작업 브랜치에만 있고 main에는 아직 반영되지 않은 상태.
2. **TASK_TREE 정리 + DONE 이력화**: 라운드 루트 노드의 모든 자식이 완료(`[x]` 또는 `[-]`)됐으면 [TASK_TREE.md](../_docs/_worklog/TASK_TREE.md) 갱신 규칙대로 트리에서 제거하고 [DONE.md](../_docs/_worklog/DONE.md)에 이력 기록. 트리에는 활성·대기 작업만 남긴다.
3. **사용자 검토 통과**: 변경 요약(브랜치명·변경 파일 수·핵심 변경 한 단락 + 짚어둘 점)을 사용자에게 보고하고 결정 신호를 받는다. AI 자율 판단으로 머지하지 않는다.
   - 사용자 결정 갈래: (a) OK·머지 진행 / (b) 추가 정정 (어느 부분을 어떻게 정정할지 명시 받고 추가 commit으로 처리 후 다시 본 단계 반복) / (c) 롤백 (브랜치 폐기).
   - "사용자 검토 통과" 시그널은 명시적이어야 한다. 사용자가 침묵하거나 다음 작업을 시작하는 것을 OK로 추론하지 않는다.
   - 단발 작업·메타 정정 한 줄·오탈자 같이 이 흐름 자체를 적용하지 않는 작은 변경은 본 단계도 면제 (cf. "단발 작업과의 경계").
4. **main 머지**: `git checkout main && git merge <branch> --no-ff`. `--no-ff`로 단위 작업 단위 머지 commit이 명시적으로 git log에 남도록. 머지 commit message는 `Merge <branch>: <라운드 요약> (사용자 검토 통과)` 형태가 권장 — git log에서 자율 머지/검토 머지 구분 가능.
5. **푸시·정리**: `git push origin main` + 작업 브랜치 삭제 (`git branch -d <branch>` + 원격에도 push했다면 `git push origin --delete <branch>`).

## 단발 작업과의 경계

자료 1건·메타 정정 한 줄·오탈자 수정 같은 단발 작업은 본 흐름을 적용하지 않고 main 직접 commit으로도 OK. 어느 쪽인지 모호하면 단위 작업 흐름이 안전(끊겨도 회복, main 단일 commit은 누적 시 정리 비용↑).

판별 휴리스틱:
- 변경이 한 commit에 자연스럽게 담기는가, 아니면 여러 commit으로 분해되는가?
- 작업이 한 세션 안에 끝나는가, 아니면 여러 세션에 걸칠 가능성이 있는가?
- TASK_TREE에 자식 노드 여럿이 등재될 만한가?
한 답이 단위 작업 쪽이면 본 흐름을 따른다.

## 끊김·재진입

세션이 끊긴 뒤 다른 세션으로 재진입할 때:
- STATUS의 "현재 단계"·TASK_TREE의 라운드 루트가 작업 컨텍스트의 1차 입구.
- git log + 작업 브랜치 위치로 어디까지 진행됐는지 확인.
- 시작점 commit 이후의 commit들을 훑어 진행 양상 복원.
- 마지막 commit과 STATUS 묘사가 어긋나면 그 사이에 끊김이 있던 것 — 어긋난 부분을 정정 후 이어가기.
