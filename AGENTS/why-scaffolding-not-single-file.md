# 왜 워크스페이스 scaffolding으로 정보를 분류하는가

## 문제

정보를 단일 파일이나 소수의 포괄적 파일에 모으면:

- 시간이 갈수록 비대해져 AI가 세션 시작 시 전체를 읽어야 한다. 토큰 소진.
- 분류 없이 선형으로 쌓이면 필요한 정보를 찾기 어렵다 (인간도 AI도).
- 포괄적 이름의 파일은 무제한적으로 길어지는 함정에 빠진다.

## 해결

워크스페이스를 처음 세팅할 때 scaffolding 폴더 구조를 만들어, 정보의 종류에 따라 위치가 결정되게 한다. cf. [workspace-and-project-structure.md](workspace-and-project-structure.md)

각 scaffolding 폴더는 디스크립터(AGENTS.md)를 두어, 디스크립터만 먼저 읽고 필요한 문서만 선택적으로 여는 방식으로 토큰 소진을 억제한다. "어떤 문서가 존재하는가"는 파일시스템(자기설명적 파일명)이 SSOT(single source of truth)다.

개별 문서는:
- 파일명이 내용을 특정할 수 있는 구체적 이름이어야 한다.
- 하나의 문서는 하나의 관심사만 다룬다.
- 단일 문서가 커지면 하위 문서로 분할한다.

## 기각안
- 단일 파일 + 목차: 목차로 탐색성은 개선되나 세션 시작 시 전체를 읽어야 하는 토큰 소진 문제는 그대로다.
- 분류 없는 다중 파일: 위치 규칙이 없으면 같은 정보가 어디에 있을지 예측할 수 없어 탐색과 배치 판단이 매번 재발생한다.

## 전제와 재검토 트리거
- 전제: "세션 시작 시 문서를 읽는 주체(AI)의 컨텍스트 예산이 유한하고, 전체 로드는 낭비다."
- 재검토 트리거: 진입 주체가 전체를 상시 로드해도 무해할 만큼 컨텍스트 예산이 커지면 디스크립터 선택 로드 구조의 이득이 줄어드므로 재검토한다.
