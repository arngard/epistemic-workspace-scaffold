# 단위 작업 운영 흐름

## 단위 작업이란

용어 정의는 [_docs/_ontology/unit-task.md](../_docs/_ontology/unit-task.md) 참조 — 의미적으로 atomic한 작업 단위, 작업 트리 한 노드에 대응. 본 문서는 그 단위 작업의 운영 흐름을 다룬다.

다음 중 하나에 해당하면 단위 작업으로 다룬다 (cf. ontology 정의 — 의미 atomic 단위):

- 자료 N건 묶음 ingest
- 메타 결정·정책 변경 (예: 갱신 규칙 신설, 카테고리 재편)
- 큰 마이그레이션·구조 재편 (예: 폴더 분리, 파일 일괄 리네임)
- 도메인 모듈 1건 신설 (예: 새 sub-INDEX 도입과 첫 배치)
- 컨텍스트가 한 세션을 넘어갈 가능성이 큰 작업

단발 한 줄 수정·자료 1건 단편 ingest는 단위 작업이 아니라 자율 사이클의 자잘한 갱신에 가깝다. 모호 사례는 단위 작업 흐름이 안전 — 끊겼을 때 회복 비용이 단발 처리보다 작다.

직후의 정정 commit은 별도 단위 작업이 아니라 **앞 작업과 같은 단위 작업의 일부**다 (의미가 같으면 한 묶음). 단위 작업의 경계는 commit·브랜치 단위가 아니라 **의미 구분 단위**.

## 진입 흐름 (단위 작업 시작)

순서를 지킨다. 한 단계도 건너뛰지 않는다.

1. **작업 트리 점검·노드 결정**: 새 작업 시작 전 [TASK_TREE.md](../_docs/_worklog/TASK_TREE.md)를 보고 어느 노드에 대한 작업인지 정한다. 노드가 없으면 신설. 단위 작업 깊이를 정함 — 그 노드를 통째로 한 단위 작업으로 잡을지, 자식들을 별도 단위 작업으로 쪼갤지 결정. TBD 환경에서는 어느 노드에 대한 브랜치인지가 브랜치명만으로는 결정 안 되므로 본 단계가 누락되면 단위 작업의 의미 단위가 흐려진다 (cf. [_docs/_ontology/unit-task.md](../_docs/_ontology/unit-task.md) "운영 권장").
2. **작업 브랜치 생성** (cf. [branch-strategy.md](branch-strategy.md)): `<type>/YYMMDD-<짧은-설명>` 형식. 1단계에서 결정된 단위 작업의 슬러그를 짧은 설명으로 사용. 단위 작업마다 격리된 commit 라인을 확보 — 도중 컨텍스트 압축·세션 끊김이 일어나도 작업 라인 추적이 깔끔하다.
3. **워크로그 등재**: `_docs/_worklog/STATUS.md`의 "현재 단계" 갱신 (작업 내용 + 시작 일시). `TASK_TREE.md`의 해당 노드를 진행 중(`[/]`)으로 표시. 1단계에서 신설한 노드면 자식들도 같이 등재. STATUS·TASK_TREE는 다음 세션 진입자가 작업 컨텍스트를 복원하는 1차 입구다.
4. **시작점 commit**: 워크로그 등재만 담은 commit으로 작업 시작점을 git log에 마킹. `--allow-empty` 라도 만들어 두면 작업 브랜치 base와 첫 실작업 사이에 명확한 anchor가 생긴다. (워크로그 변경이 있으면 그걸 포함, 없으면 빈 commit.)

## 진행 흐름 (작업 중)

- 처리 단위(자료 1건·하위 단계 1개) 완료마다 또는 2-3건 단위로 commit. 컨텍스트 부담이 큰 작업은 더 작은 단위로 끊는다.
- TASK_TREE의 처리 단위 노드를 `[x]`로 갱신. 새로 드러난 항목은 즉시 추가.
- 컨텍스트 압축이 임박하면 STATUS의 "현재 단계"를 다음 세션이 이어받을 수 있는 형태로 다듬는다 (어디까지 진행, 다음에 무엇).
- 작업 브랜치는 자유롭게 push 해도 되고 안 해도 된다. 다만 세션이 끊길 가능성이 있으면 push 해 두는 게 안전.

## 종료 흐름 (단위 작업 끝)

순서를 지킨다.

1. **마무리 commit**: 작업 마지막 정리(미캡처 보강·표준 패턴 추가 학습 사항 반영·STATUS 마지막 작업 일시 갱신 등)를 담은 commit. 이 시점까지의 모든 작업은 작업 브랜치에만 있고 main에는 아직 반영되지 않은 상태.
2. **TASK_TREE 정리 + DONE 이력화**: 단위 작업 노드의 모든 자식이 완료(`[x]` 또는 `[-]`)됐으면 [TASK_TREE.md](../_docs/_worklog/TASK_TREE.md) 갱신 규칙대로 트리에서 제거하고 [DONE.md](../_docs/_worklog/DONE.md)에 이력 기록. 트리에는 활성·대기 작업만 남긴다.
3. **사용자 검토 통과**: 변경 요약(브랜치명·변경 파일 수·핵심 변경 한 단락 + 짚어둘 점)을 사용자에게 보고하고 결정 신호를 받는다. AI 자율 판단으로 머지하지 않는다.
   - 사용자 결정 갈래: (a) OK·머지 진행 / (b) 추가 정정 (어느 부분을 어떻게 정정할지 명시 받고 추가 commit으로 처리 후 다시 본 단계 반복) / (c) 롤백 (브랜치 폐기).
   - "사용자 검토 통과" 시그널은 명시적이어야 한다. 사용자가 침묵하거나 다음 작업을 시작하는 것을 OK로 추론하지 않는다.
   - 단발 작업·메타 정정 한 줄·오탈자 같이 이 흐름 자체를 적용하지 않는 작은 변경은 본 단계도 면제 (cf. "단발 작업과의 경계").
4. **main 머지**: `git checkout main && git merge <branch> --no-ff`. `--no-ff`로 단위 작업 단위 머지 commit이 명시적으로 git log에 남도록. 머지 commit message는 `Merge <branch>: <단위 작업 요약> (사용자 검토 통과)` 형태가 권장 — git log에서 자율 머지/검토 머지 구분 가능.
5. **푸시·정리**: `git push origin main` + 작업 브랜치 삭제 (`git branch -d <branch>` + 원격에도 push했다면 `git push origin --delete <branch>`).

## 단발 작업과의 경계

자료 1건·메타 정정 한 줄·오탈자 수정 같은 단발 작업은 본 흐름을 적용하지 않고 main 직접 commit으로도 OK. 어느 쪽인지 모호하면 단위 작업 흐름이 안전(끊겨도 회복, main 단일 commit은 누적 시 정리 비용↑).

판별 휴리스틱:
- 변경이 한 commit에 자연스럽게 담기는가, 아니면 여러 commit으로 분해되는가?
- 작업이 한 세션 안에 끝나는가, 아니면 여러 세션에 걸칠 가능성이 있는가?
- TASK_TREE에 자식 노드 여럿이 등재될 만한가?
한 답이 단위 작업 쪽이면 본 흐름을 따른다.

## PR 운영

본 절은 단위 작업 종료 흐름에서 main 머지 전에 PR을 거치는 모든 경우에 적용된다 (사용자 검토 통과를 PR 검토로 받는 흐름·routines 자동 트리거 PR 패턴 등).

### 본문 형식

PR 본문은 다음 두 섹션을 갖춘다. 추가 섹션은 자유.

- `## 배경`: 이전의 관련 PR·issue·메일·채팅 결정, 본 PR을 트리거한 상황 설명, 리뷰어가 본 PR을 이해하는 데 필요한 메타 정보. 본 섹션이 비면 리뷰어는 변경 내용만 보고 의도와 맥락을 추론해야 하며, 리뷰 품질이 떨어진다.
- `## 변경`: 변경 사항의 나열. 단순 PR은 bullet list로 충분, 분량이 크거나 영역이 갈리면 `### 소제목`별로 구분.

추가 섹션은 자유 — 검토 체크리스트·다음 단위 작업으로 분리한 사항·동행 chore 안내 등은 본 PR 운영 흐름에 자연스럽게 끼워 넣는다.

### 속성 할당

PR 생성 시 다음 속성을 적절히 할당한다.

- Assignees: 본 PR의 작업자·검토 책임자. AI가 PR을 생성한 경우 작업 의뢰한 사용자를 assign.
- Labels: 변경 종류 라벨(chore·docs·feature·fix 등) + 영역 라벨(레포별 도메인에 따라). 라벨 풀이 비어 있으면 본 PR이 첫 라벨을 만들 수도 있다 — 향후 분류 비용 회수를 위해 작은 라벨이라도 등재한다.
- Milestone·Projects: 레포 운영 흐름에 따라 선택. 본 워크플로우는 강제하지 않는다.

### 리뷰 코멘트 응답 방식

PR에 달린 인라인 리뷰 코멘트(사용자 본인·다른 개발자·자동 리뷰 봇 — Gemini Code Assist 등)에 응답할 때는 **그 코멘트의 스레드 reply**로 단다. PR 본체에 새 issue-level 코멘트를 다는 방식(`gh pr comment`)은 리뷰 스레드와 별개의 평면에 놓여 두 평면의 문맥이 섞이고 이후 추적이 어려워진다.

도구별 호출 형태:

- GitHub Web UI: 해당 리뷰 코멘트의 "Reply" 입력란 사용.
- gh CLI: `gh api repos/<owner>/<repo>/pulls/<num>/comments/<comment_id>/replies -f body="..."`
  - `<comment_id>`는 리뷰 코멘트 URL 끝 `#discussion_r<숫자>`의 숫자 부분. `gh api repos/<owner>/<repo>/pulls/<num>/comments`로도 조회 가능.
- 금지: `gh pr comment <num> --body "..."` — 본 명령은 issue-level 코멘트를 만든다. 본 PR에 새 일반 코멘트를 의도적으로 단다는 게 명백한 경우(머지 알림·후속 작업 안내 등)에만 사용.

잘못 단 경우 정정: 잘못된 코멘트 ID 조회(`gh api repos/<owner>/<repo>/issues/<num>/comments`) 후 삭제(`gh api repos/<owner>/<repo>/issues/comments/<id> -X DELETE`), 그 뒤 스레드 reply로 재발행. 인라인 리뷰 코멘트 ID는 별도 엔드포인트 — `gh api repos/<owner>/<repo>/pulls/<num>/comments` 또는 코멘트 URL 끝 `#discussion_r<숫자>`.

본 정책은 자동 리뷰 봇 응답에도 똑같이 적용 — 봇 코멘트라도 사용자가 같은 PR을 나중에 다시 볼 때 스레드 문맥이 깨져 있으면 디버깅 비용이 커진다.

### 명시적 멘션 의무

리뷰 코멘트에 응답할 때 본인이 코멘트를 단 사실만으로 스레드 참여자가 알림을 받으리라고 가정하지 않는다. 응답 본문에 듣는 대상을 `@username`으로 명시적으로 멘션한다 — 자동 리뷰 봇도 동일 적용(예: `@gemini-code-assist`). 멘션이 없으면 GitHub 알림 채널이 닿지 않거나, 닿더라도 누구에게 답하는 코멘트인지 모호해 스레드 가독성이 떨어진다.

복수 대상 멘션은 첫 줄에 모아둔다 — 본 코멘트가 누구를 향한 것인지를 본문 진입과 동시에 보이게.

### AI 시그니처 의무

적용 범위: AI가 사용자의 토큰·계정으로 외부 협업 시스템(GitHub PR 본문·이슈·댓글·리뷰 reply 등)에 글을 쓰는 경우에 한정. 채팅 응답·세션 내 의사 소통·로컬 명령 출력에는 적용하지 않는다 — 그 경로는 발화자가 AI임이 1:1 대화 맥락에서 자명해 시그니처가 정보를 추가하지 않는다.

형식:

```
Generated with {AI 이름}
```

예: `Generated with Claude Code`, `Generated with Gemini Code Assist`.

근거: 외부 협업 시스템에서는 사용자 토큰으로 작성된 글이 사용자의 직접 행동인지 AI 자동 행동인지를 외부 관찰자(다른 협업자·리뷰 봇·사후 본인)가 식별할 수 없다. 시그니처가 그 식별 단서가 된다. 누가 무엇을 결정·작성했는지가 흐려지면 이후 정정·검토·인용 비용이 커진다.

도구별 적용:

- PR 본문: 일부 AI(Claude Code 등)는 자동으로 footer에 추가한다. 자동 추가 여부를 확인하고, 없으면 명시 추가.
- 댓글·리뷰 코멘트 reply: 자동 추가 없는 경우가 많아 명시 의무. 본문 마지막 한 줄로 단다.

## 끊김·재진입

세션이 끊긴 뒤 다른 세션으로 재진입할 때:
- STATUS의 "현재 단계"·TASK_TREE의 진행 중 단위 작업 노드가 작업 컨텍스트의 1차 입구.
- git log + 작업 브랜치 위치로 어디까지 진행됐는지 확인.
- 시작점 commit 이후의 commit들을 훑어 진행 양상 복원.
- 마지막 commit과 STATUS 묘사가 어긋나면 그 사이에 끊김이 있던 것 — 어긋난 부분을 정정 후 이어가기.
