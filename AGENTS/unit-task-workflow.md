# 단위 작업 운영 흐름

## 단위 작업이란

용어 정의는 [_docs/_ontology/unit-task.md](../_docs/_ontology/unit-task.md) 참조 — 의미적으로 atomic한 작업 단위, 작업 트리 한 노드에 대응. 본 문서는 그 단위 작업의 운영 흐름을 다룬다.

다음 중 하나에 해당하면 단위 작업으로 다룬다 (cf. ontology 정의 — 의미 atomic 단위):

- 자료 N건 묶음 ingest
- 메타 결정·정책 변경 (예: 갱신 규칙 신설, 카테고리 재편)
- 큰 마이그레이션·구조 재편 (예: 폴더 분리, 파일 일괄 리네임)
- 도메인 모듈 1건 신설 (예: 새 sub-INDEX 도입과 첫 배치)
- 컨텍스트가 한 세션을 넘어갈 가능성이 큰 작업

단발 한 줄 수정·자료 1건 단편 ingest는 단위 작업이 아니라 자율 사이클의 자잘한 갱신에 가깝다. 모호 사례는 단위 작업 흐름이 안전 — 끊겼을 때 회복 비용이 단발 처리보다 작다.

직후의 정정 커밋은 별도 단위 작업이 아니라 **앞 작업과 같은 단위 작업의 일부**다 (의미가 같으면 한 묶음). 단위 작업의 경계는 커밋·브랜치 단위가 아니라 **의미 구분 단위**.

## 진입 흐름 (단위 작업 시작)

순서를 지킨다. 한 단계도 건너뛰지 않는다.

1. **작업 트리 점검·노드 결정 + 관련 절차 문서 재독**: 새 작업 시작 전 [TASK_TREE.md](../_docs/_worklog/TASK_TREE.md)를 보고 어느 노드에 대한 작업인지 정한다. 노드가 없으면 신설. 단위 작업 깊이를 정함 — 그 노드를 통째로 한 단위 작업으로 잡을지, 자식들을 별도 단위 작업으로 쪼갤지 결정. TBD 환경에서는 어느 노드에 대한 브랜치인지가 브랜치명만으로는 결정 안 되므로 본 단계가 누락되면 단위 작업의 의미 단위가 흐려진다 (cf. [_docs/_ontology/unit-task.md](../_docs/_ontology/unit-task.md) "운영 권장").

    노드 결정 직후 **관련 절차 문서를 재독**한다. 본 워크스페이스의 운영 절차 문서들 — `_docs/_architecture/ingest-workflow.md`·`_docs/_architecture/conventions.md`·자식 워크스페이스의 도메인 절차 문서 등 — 중 본 단위 작업의 영역에 해당하는 것을 처음부터 다시 읽는다. 자식 워크스페이스는 진입점 AGENTS.md에 "요청 의도 → 절차 문서 매핑" 표가 있을 수 있으니 그 표를 우선 활용한다.

    재독 의무의 근거: 검색·인제스트·리포트 작성처럼 한 단위 작업의 컨텍스트가 큰 흐름에서는 절차 문서의 세부 규약을 자기 기억으로 단언하다가 소소한 위반이 누적된다. 누적 정합성 부채의 빈번한 원천이 본 패턴이다. 본 단계의 재독은 단순 형식이 아니라 본 단위 작업의 1차 가드레일이다.

    재독 면제 조건: 본 단위 작업이 메타 결정·운영 정책 변경 자체이고, 그 변경 대상 문서를 이미 같은 단위에서 직접 손대고 있는 경우(예: ingest-workflow.md를 본 단위에서 수정하는 작업은 본 문서를 이미 읽는 셈). 일반 도메인 작업(인제스트·인지해석·리포트 등)에는 면제 없음.
2. **작업 브랜치 생성** (cf. [branch-strategy.md](branch-strategy.md)): `<type>/YYMMDD-<짧은-설명>` 형식. 1단계에서 결정된 단위 작업의 슬러그를 짧은 설명으로 사용. 단위 작업마다 격리된 커밋 라인을 확보 — 도중 컨텍스트 압축·세션 끊김이 일어나도 작업 라인 추적이 깔끔하다.

    종료 흐름의 기본이 풀리퀘 타입(아래 4-1)이므로 main 작업 트리와 동시 작업·전환이 필요한 경우 워크트리(`git worktree add`)로 작업 공간 분리를 권장한다 — main 트리와 독립된 디렉토리에서 작업하다 종료 후 `git worktree remove`로 정리. 브랜치 관리자 타입으로 진행하는 작업에도 적용 가능.
3. **워크로그 등재**: `_docs/_worklog/STATUS.md`의 "현재 단계" 갱신 (작업 내용 + 시작 일시). `TASK_TREE.md`의 해당 노드를 진행 중(`[/]`)으로 표시. 1단계에서 신설한 노드면 자식들도 같이 등재. STATUS·TASK_TREE는 다음 세션 진입자가 작업 컨텍스트를 복원하는 1차 입구다.
4. **시작점 커밋**: 워크로그 등재만 담은 커밋으로 작업 시작점을 git log에 마킹. `--allow-empty` 라도 만들어 두면 작업 브랜치 base와 첫 실작업 사이에 명확한 anchor가 생긴다. (워크로그 변경이 있으면 그걸 포함, 없으면 빈 커밋.)

## 진행 흐름 (작업 중)

- 처리 단위(자료 1건·하위 단계 1개) 완료마다 또는 2-3건 단위로 커밋. 컨텍스트 부담이 큰 작업은 더 작은 단위로 끊는다.
- TASK_TREE의 처리 단위 노드를 `[x]`로 갱신. 새로 드러난 항목은 즉시 추가.
- 컨텍스트 압축이 임박하면 STATUS의 "현재 단계"를 다음 세션이 이어받을 수 있는 형태로 다듬는다 (어디까지 진행, 다음에 무엇).
- 작업 브랜치는 자유롭게 푸시 해도 되고 안 해도 된다. 다만 세션이 끊길 가능성이 있으면 푸시 해 두는 게 안전.

## 종료 흐름 (단위 작업 끝)

순서를 지킨다.

1. **마무리 커밋**: 작업 마지막 정리(미캡처 보강·표준 패턴 추가 학습 사항 반영·STATUS 마지막 작업 일시 갱신 등)를 담은 커밋. 이 시점까지의 모든 작업은 작업 브랜치에만 있고 main에는 아직 반영되지 않은 상태.

    마무리 커밋 시점의 워크로그(STATUS·TASK_TREE·DONE)는 **본 단위 작업이 main 머지된 이후의 상태를 미리 기록**한다. 본 시점의 작업 브랜치는 아직 main에 머지되지 않았지만, 머지 직후 그대로 정합 상태가 되어야 별도의 워크로그 정정 커밋이 안 생긴다 — PR이 오래 open 상태로 머무는 경우에도 동일.
2. **TASK_TREE 정리 + DONE 이력화**: 단위 작업 노드의 모든 자식이 완료(`[x]` 또는 `[-]`)됐으면 [TASK_TREE.md](../_docs/_worklog/TASK_TREE.md) 갱신 규칙대로 트리에서 제거하고 [DONE.md](../_docs/_worklog/DONE.md)에 이력 기록. 트리에는 활성·대기 작업만 남긴다.
3. **사용자 검토 통과**: 변경 요약(브랜치명·변경 파일 수·핵심 변경 한 단락 + 짚어둘 점)을 사용자에게 보고하고 결정 신호를 받는다. AI 자율 판단으로 머지하지 않는다.
   - 사용자 결정 갈래: (a) OK·머지 진행 / (b) 추가 정정 (어느 부분을 어떻게 정정할지 명시 받고 추가 커밋으로 처리 후 다시 본 단계 반복) / (c) 롤백 (브랜치 폐기).
   - "사용자 검토 통과" 시그널은 명시적이어야 한다. 사용자가 침묵하거나 다음 작업을 시작하는 것을 OK로 추론하지 않는다.
   - 시그널 형태는 채팅 내 명시 승인("머지해" 등 본 시점 직접 표명)과 PR Approve가 등가다 — 두 형태 모두 동일 diff가 main에 편입되는 결정 신호로 본다. 어느 형태로 받을지는 4단계의 타입 분기로 결정된다.
   - 단발 작업·메타 정정 한 줄·오탈자 같이 이 흐름 자체를 적용하지 않는 작은 변경은 본 단계도 면제 (cf. "단발 작업과의 경계").
4. **main 머지** — 타입 분기: 본 단계는 두 타입 중 하나로 진행된다.

    종료 흐름 4단계의 두 분기는 **풀리퀘 타입**(PR 흐름)과 **브랜치 관리자 타입**(직접 머지)으로 부른다. 'A/B' 같은 추상 라벨 대신 의미가 드러나는 표현을 사용한다 — 이후 외부 인용(커밋 메시지·DONE·STATUS·다른 문서)에서도 본 명명을 따른다.

    기본은 풀리퀘 타입. 브랜치 관리자 타입으로 진행하려면 사용자의 명시적 허용·지시가 필요하다 — "그냥 머지해", "메인 푸시 OK", "PR 생략" 같이 PR 단계 생략 의도가 드러나는 표현이 해당된다.

    두 타입의 분기점은 본 단계(머지 형태)뿐이다. 진입 흐름·진행 흐름·종료 흐름의 1·2·3·5단계는 타입 공통이며, **작업 브랜치도 두 타입 공통**이다 — 브랜치 관리자 타입은 PR 검토 경유를 생략할 뿐 작업 브랜치 자체를 생략하지 않는다. main 직접 커밋으로도 충분한 단발 변경은 본 흐름의 대상이 아니다 (cf. [단발 작업과의 경계](#단발-작업과의-경계)).

    routines·cron 자동 트리거는 풀리퀘 타입을 강제한다 — 자율 진행에는 사람 리뷰가 필수. 타입 결정이 모호하면 사용자에게 다시 확인한다. AI 자율로 타입을 결정하지 않는다.

    ### 4-1. 풀리퀘 타입 (PR 흐름)

    1. PR 생성 — 본문 형식·자기 검토·속성 할당·시그니처 등 규약은 [PR 운영](#pr-운영) 절 적용.
    2. 리뷰 대기 + [능동 리뷰 확인](#제출-후-능동-리뷰-확인).
    3. 리뷰 코멘트가 있으면 응답·정정 ([리뷰 코멘트 응답 방식](#리뷰-코멘트-응답-방식)·[명시적 멘션 의무](#명시적-멘션-의무)·[AI 시그니처 의무](#ai-시그니처-의무) 적용).
    4. 사용자 PR Approve 시그널 — 본 시그널이 3단계 "사용자 검토 통과"의 PR 형태.
    5. main 머지 (GitHub UI 또는 `gh pr merge --merge` 같은 CLI). origin 푸시가 머지에 포함됨.
    6. 작업 브랜치 정리 (`gh pr merge --delete-branch` 또는 별도 `git push origin --delete <branch>` + 로컬 `git branch -d <branch>`). 워크트리를 사용한 경우 로컬 `git branch -d` **이전**에 `git worktree remove <경로>`로 작업 워크트리부터 정리한다 — 서브모듈을 포함하는 워크트리는 `--force`가 필요할 수 있다. 정리 후 [잔존 점검](#잔존-점검) 적용.

    ### 4-2. 브랜치 관리자 타입 (직접 머지)

    ```bash
    git checkout main
    git merge <branch> --no-ff
    git push origin main
    git branch -d <branch>
    git push origin --delete <branch>   # 원격에도 푸시했다면
    ```

    `--no-ff`로 단위 작업 단위 머지 커밋이 git log에 명시적으로 남도록. 머지 커밋 메시지는 `Merge <branch>: <단위 작업 요약> (사용자 검토 통과)` 형태가 권장 — git log에서 직접 머지·PR 머지 구분 가능.

    워크트리를 사용한 경우 위 시퀀스의 `git branch -d <branch>` **이전**에 `git worktree remove <경로>`로 작업 워크트리부터 정리한다 — 워크트리가 브랜치를 점유한 상태에서는 `git branch -d`가 실패한다. 서브모듈을 포함하는 워크트리는 잔존 git 메타데이터로 인해 `--force`가 필요할 수 있다. 정리 후 [잔존 점검](#잔존-점검) 적용.

    환경 안전망이 main 푸시를 차단하는 경우(일부 AI harness의 "main 직접 푸시 금지" 규약 등), 우회 편법(PR API 엔드포인트를 통한 머지 등)을 자율로 쓰지 않는다. 차단 사실을 사용자에게 보고하고 처리 결정을 요청한다 — 브랜치 관리자 타입 재시도·풀리퀘 타입 전환·보류 모두 사용자 결정 영역이며, AI가 자율로 풀리퀘 타입으로 전환하지도 않는다.

    ### 잔존 점검

    4-1·4-2 모두에 공통. 머지·푸시·브랜치 삭제 + 워크트리 정리 직후 다음을 확인한다.

    - `git worktree list` — 작업 워크트리가 목록에 남지 않았는지.
    - `git branch -a` — 로컬·원격 모두에 작업 브랜치가 남지 않았는지 (원격 추적 브랜치는 `git fetch --prune`까지 동반).

    잔존이 발견되면 별도 단위 작업이 아니라 본 단위 작업의 후속으로 즉시 정리한다 — 다음 단위 작업으로 넘기지 않는다.

5. **서브모듈 커밋 포인트 재귀 정합화** (해당하는 경우): 본 레포가 다른 레포의 서브모듈로 등록돼 있다면, 그 상위 레포에서 서브모듈 pointer를 본 단위 작업 종료 시점의 main HEAD로 갱신한다 — `cd <상위 레포> && git add <서브모듈 경로> && git commit -m "..." && git push origin main`(`push`까지 한 단위). 본 정합화는 재귀적이다 — 상위 레포가 다시 더 상위 레포의 서브모듈이면 거기까지 같은 절차를 반복한다. 트리의 모든 커밋 포인트가 정합 상태에 도달해야 단위 작업이 진짜 끝. 본 레포 입장에서는 자기가 등재된 곳만 챙기면 충분하지만, 본 레포가 다시 다른 자식 변경을 흡수하는 상위 노드라면 자식 단위 작업이 같은 원칙으로 본 레포까지 정렬을 올린다 — 양방향이 아니라 "본인이 변경되면 본인을 가리키는 상위까지 위로 흐른다"의 단방향 재귀. 부모 레포에 변경 노트 누적 문서(예: `UPGRADING.md`)가 있다면 자식 hash 추기까지 같은 단위에 묶는다. 본 단계가 누락되면 부모 레포가 옛 자식 커밋을 가리키는 상태로 부정합 — 자식 sync는 끝나도 부모는 끝나지 않은 상태가 된다. 검증: 정합화 트리의 모든 노드에서 `git submodule status`의 `+` 표시 없음 + `git diff --submodule` 비어 있음. 본 레포가 어디에 등재됐는지는 본 레포 안에서 직접 보이지 않으므로, 부모 레포 자체 문서(예: README의 "자식 sync 라운드 종료" 같은 절차)에서 본 단계의 구체 시점·도구·문서 형식을 별도 관리한다.

## 단발 작업과의 경계

자료 1건·메타 정정 한 줄·오탈자 수정 같은 단발 작업은 본 흐름을 적용하지 않고 main 직접 커밋으로도 OK. 어느 쪽인지 모호하면 단위 작업 흐름이 안전(끊겨도 회복, main 단일 커밋은 누적 시 정리 비용↑).

판별 휴리스틱:
- 변경이 한 커밋에 자연스럽게 담기는가, 아니면 여러 커밋으로 분해되는가?
- 작업이 한 세션 안에 끝나는가, 아니면 여러 세션에 걸칠 가능성이 있는가?
- TASK_TREE에 자식 노드 여럿이 등재될 만한가?
한 답이 단위 작업 쪽이면 본 흐름을 따른다.

## PR 운영

본 절은 종료 흐름 4단계의 풀리퀘 타입(PR 흐름)에 적용되는 PR 본문·검토·응답 규약이다. 브랜치 관리자 타입으로 진행한 단위 작업이라도 이후 별도 PR·이슈·리뷰 코멘트 대응이 필요한 시점에는 일부 규약(시그니처·리뷰 응답 방식 등)이 적용된다.

### 본문 형식

PR 본문은 다음 두 섹션을 갖춘다. 추가 섹션은 자유.

- `## 배경`: 이전의 관련 PR·issue·메일·채팅 결정, 본 PR을 트리거한 상황 설명, 리뷰어가 본 PR을 이해하는 데 필요한 메타 정보. 본 섹션이 비면 리뷰어는 변경 내용만 보고 의도와 맥락을 추론해야 하며, 리뷰 품질이 떨어진다.
- `## 변경`: 변경 사항의 나열. 단순 PR은 bullet list로 충분, 분량이 크거나 영역이 갈리면 `### 소제목`별로 구분.

추가 섹션은 자유 — 검토 체크리스트·다음 단위 작업으로 분리한 사항·동행 chore 안내 등은 본 PR 운영 흐름에 자연스럽게 끼워 넣는다.

### 제출 전 자기 검토

PR 생성 직전에 본인이 점검할 항목:

- **부모 브랜치 머지 시 더 수정 필요 없는 상태인가?** 본 PR의 작업 완료를 의미하는 워크로그 갱신(STATUS·TASK_TREE·DONE)은 본 PR의 커밋에 포함되어 있어야 한다. 본 PR이 머지된 뒤에 "이 PR과 의미적으로 합쳐져야 하는 추가 작업"이 남으면 단위 작업 분리가 어긋난다. 머지 후 정정 단위가 별도로 필요할 만한 상태는 본 PR 안에서 흡수해 끝낸다.
- **보는 사람에게 충분한 설명인가?** `## 배경`·`## 변경` 본문이 PR을 처음 보는 외부 관찰자(다른 협업자·리뷰 봇·사후 본인)에게 의도·트리거·변경 영역을 충분히 전달하는지 점검. 관련 PR 번호·결정 흐름·동행 chore 안내가 본문 안에 있어야 한다.

본 검토는 객관성을 위해 서브에이전트(예: code-review·report-reviewer 등)에 위임할 수 있다. AI가 본인 작업을 본인이 검토하면 사각지대가 남기 때문이다.

### 속성 할당

PR 생성 시 다음 속성을 적절히 할당한다.

- Assignees: 본 PR의 작업자·검토 책임자. AI가 PR을 생성한 경우 작업 의뢰한 사용자를 assign.
- Labels: 변경 종류 라벨(chore·docs·feature·fix 등 — [branch-strategy.md](branch-strategy.md) 브랜치 타입과 통일) + 영역 라벨(레포별 도메인에 따라). 라벨 풀이 비어 있으면 본 PR이 첫 라벨을 만들 수도 있다 — 향후 분류 비용 회수를 위해 작은 라벨이라도 등재한다.
- Milestone·Projects: 레포 운영 흐름에 따라 선택. 본 워크플로우는 강제하지 않는다.

### 리뷰 코멘트 응답 방식

PR에 달린 인라인 리뷰 코멘트(사용자 본인·다른 개발자·자동 리뷰 봇 — Gemini Code Assist 등)에 응답할 때는 **그 코멘트의 스레드 reply**로 단다. PR 본체에 새 issue-level 코멘트를 다는 방식(`gh pr comment`)은 리뷰 스레드와 별개의 평면에 놓여 두 평면의 문맥이 섞이고 이후 추적이 어려워진다.

도구별 호출 형태:

- GitHub Web UI: 해당 리뷰 코멘트의 "Reply" 입력란 사용.
- gh CLI: `gh api repos/<owner>/<repo>/pulls/<num>/comments/<comment_id>/replies -f body="..."`
  - `<comment_id>`는 리뷰 코멘트 URL 끝 `#discussion_r<숫자>`의 숫자 부분. `gh api repos/<owner>/<repo>/pulls/<num>/comments`로도 조회 가능.
- 금지: `gh pr comment <num> --body "..."` — 본 명령은 issue-level 코멘트를 만든다. 본 PR에 새 일반 코멘트를 의도적으로 단다는 게 명백한 경우(머지 알림·후속 작업 안내 등)에만 사용.

잘못 단 경우 정정: 잘못된 코멘트 ID 조회(`gh api repos/<owner>/<repo>/issues/<num>/comments`) 후 삭제(`gh api repos/<owner>/<repo>/issues/comments/<id> -X DELETE`), 그 뒤 스레드 reply로 재발행. 인라인 리뷰 코멘트 ID는 별도 엔드포인트 — `gh api repos/<owner>/<repo>/pulls/<num>/comments` 또는 코멘트 URL 끝 `#discussion_r<숫자>`.

본 정책은 자동 리뷰 봇 응답에도 똑같이 적용 — 봇 코멘트라도 사용자가 같은 PR을 나중에 다시 볼 때 스레드 문맥이 깨져 있으면 디버깅 비용이 커진다.

### 스레드 안 토론은 그 스레드가 SSOT

스레드 안에서 토론·결론·정정이 이루어졌으면 그 내용은 그 스레드를 SSOT로 본다. 같은 결론을 PR 본문·다른 코멘트·외부 문서에서 다시 풀어 설명하지 않는다. 외부 위치에서 짚어야 할 때는 해당 스레드(또는 결정 코멘트)의 URL을 가리키는 링크만 제시한다 — `cf. [해당 스레드](url)` 형태가 권장.

근거: 같은 내용을 두 평면에 적으면 SSOT 위배 + 한 평면이 갱신되고 다른 평면이 안 갱신될 때 결론 추적이 깨진다. 토론 흐름은 시간순으로 스레드 안에 형성되며 이걸 외부에 옮기면 그 흐름 자체가 손실된다.

### 명시적 멘션 의무

리뷰 코멘트에 응답할 때 본인이 코멘트를 단 사실만으로 스레드 참여자가 알림을 받으리라고 가정하지 않는다. 응답 본문에 듣는 대상을 `@username`으로 명시적으로 멘션한다 — 자동 리뷰 봇도 동일 적용(예: `@gemini-code-assist`). 멘션이 없으면 GitHub 알림 채널이 닿지 않거나, 닿더라도 누구에게 답하는 코멘트인지 모호해 스레드 가독성이 떨어진다.

복수 대상 멘션은 첫 줄에 모아둔다 — 본 코멘트가 누구를 향한 것인지를 본문 진입과 동시에 보이게.

### AI 시그니처 의무

적용 범위: AI가 사용자의 토큰·계정으로 외부 협업 시스템(GitHub PR 본문·이슈·댓글·리뷰 reply 등)에 글을 쓰는 경우에 한정. 채팅 응답·세션 내 의사 소통·로컬 명령 출력에는 적용하지 않는다 — 그 경로는 발화자가 AI임이 1:1 대화 맥락에서 자명해 시그니처가 정보를 추가하지 않는다.

형식:

```
Generated with {AI 이름}
```

예: `Generated with Claude Code`, `Generated with Gemini Code Assist`.

근거: 외부 협업 시스템에서는 사용자 토큰으로 작성된 글이 사용자의 직접 행동인지 AI 자동 행동인지를 외부 관찰자(다른 협업자·리뷰 봇·사후 본인)가 식별할 수 없다. 시그니처가 그 식별 단서가 된다. 누가 무엇을 결정·작성했는지가 흐려지면 이후 정정·검토·인용 비용이 커진다.

도구별 적용:

- PR 본문: 일부 AI(Claude Code 등)는 자동으로 footer에 추가한다. 자동 추가 여부를 확인하고, 없으면 명시 추가.
- 댓글·리뷰 코멘트 reply: 자동 추가 없는 경우가 많아 명시 의무. 본문과 한 줄 띄운 뒤 마지막에 단다.

### 신규 발행 vs edit (알림 트리거)

PR 본문·제목 edit은 GitHub 알림을 새로 트리거하지 않는 경우가 많다 — 특히 자동 리뷰 봇은 PR opened 이벤트에만 동작한다. PR 본문·제목 정정이 의미적으로 큰 경우(예: 본문이 비어 있던 상태에서 채워 넣은 경우·검토 맥락이 바뀌는 보강)는 close → reopen하거나 새 코멘트로 본문 변경 사실을 알리는 편이 안전하다. 사소한 오탈자·표현 정정은 트리거 없이 그대로 둬도 무방.

### 제출 후 능동 리뷰 확인

PR 생성·푸시 직후 검토자(자동 리뷰 봇·사용자·다른 개발자)의 리뷰가 달릴 때까지 시간차가 있다. 본 라운드를 PR 머지로 이어가려면 그 리뷰를 능동적으로 감지·처리해야 한다.

확인 방법:

- 환경이 GitHub webhook을 받을 수 있다면 webhook 트리거로 처리.
- 그렇지 않으면 PR 생성 후 약 10분의 타이머를 걸어 능동 확인. 5분은 자동 리뷰 봇(예: Gemini Code Assist)의 응답이 아직 안 들어와 빈 결과만 보고 헛 종료되는 빈도가 높았다 — 10분 정도가 1차 응답 안착 시점에 더 가깝다. 봇이 더 느린 환경이라면 사용자가 별도로 조정. 도구: `gh pr view <number> --repo <owner>/<repository> --json reviewDecision,reviews,comments` + `gh api repos/<owner>/<repository>/pulls/<number>/comments`. `reviewDecision`은 PR 전체 상태(APPROVED·CHANGES_REQUESTED·REVIEW_REQUIRED 등)를 열거형으로 제공해 머지 가능 여부 1차 판단을 빠르게 한다 — 개별 리뷰 코멘트 파싱은 [후속 단계](#리뷰가-달려-있으면-즉시-처리).

AI 도구가 사용자 턴에만 동작하는 환경(예: Claude Code의 일반 채팅 흐름)에서는 "5분 후 확인"을 자율로 보장할 수 없다 — 사용자가 다음 메시지를 보낼 때까지 AI는 정지 상태다. 따라서 다음 중 하나로 처리한다.

- 환경에 자동 wake-up 도구가 있으면(예: Claude Code의 `ScheduleWakeup`, 외부 cron 트리거 등) PR 생성 직후 그 도구로 능동 확인 턴을 예약한다.
- 자동 트리거 도구가 없으면 사용자에게 "잠시 후 PR 리뷰 확인을 트리거해 주세요"라고 명시 요청한 뒤 사용자 응답을 기다린다.

"잠시 후 자율 확인하겠다"는 표현은 턴 메커니즘 한계로 거짓 약속이 되므로 쓰지 않는다.

리뷰가 달려 있으면 즉시 처리:

- 수용 가능한 지적은 본 PR 안에서 정정 커밋 + 스레드 reply (cf. [리뷰 코멘트 응답 방식](#리뷰-코멘트-응답-방식)·[명시적 멘션 의무](#명시적-멘션-의무)·[AI 시그니처 의무](#ai-시그니처-의무) 적용).
- 반박이 필요한 지적은 근거(공식 문서·경험적 증거)를 갖춰 스레드 reply.
- 대응이 모호한 경우 해당 리뷰 스레드(또는 일반 댓글)에서 사용자(`@<owner>`)를 명시적으로 멘션하여 결정 요청. 모호한 채로 머지·롤백 결정을 AI가 단정하지 않는다.

리뷰 처리 후에도 추가 리뷰가 달릴 수 있으므로 머지 직전까지 같은 사이클을 반복.

## 끊김·재진입

세션이 끊긴 뒤 다른 세션으로 재진입할 때:
- STATUS의 "현재 단계"·TASK_TREE의 진행 중 단위 작업 노드가 작업 컨텍스트의 1차 입구.
- git log + 작업 브랜치 위치로 어디까지 진행됐는지 확인.
- 시작점 커밋 이후의 커밋들을 훑어 진행 양상 복원.
- 작업 브랜치에 발행된 open PR이 있는지 `gh pr list --head <branch>` 또는 `gh pr status`로 점검. 있으면 그 PR의 리뷰 상태가 본 단위 작업의 현재 단계 — 마무리 커밋 이후라면 [제출 후 능동 리뷰 확인](#제출-후-능동-리뷰-확인) 흐름으로 복귀.
- 마지막 커밋과 STATUS 묘사가 어긋나면 그 사이에 끊김이 있던 것 — 어긋난 부분을 정정 후 이어가기.

## 다른 세션 작업과의 격리

본 세션이 발견한 다른 세션의 산출물(작업 브랜치·워크트리·열린 PR·이슈·리뷰 코멘트 등)은 본 세션의 작업 대상이 아니면 건드리지 않는다. 머지·삭제·force-push·브랜치 정리 같은 변경 행위는 그 세션 자체에서 처리하거나 사용자의 명시 지시가 있을 때만 한다.

정리가 필요해 보여도 자율 정리하지 않는다 — 다른 세션의 작업 컨텍스트를 모르는 상태에서의 정리는 의도하지 않은 작업 손실(미머지 작업 폐기·머지 직전 상태 무력화·진행 중 PR 닫힘 등)로 이어진다. 본 세션 시점에서 "잔존"처럼 보이는 항목도 그 세션이 일부러 남겨둔 진행 중 상태일 수 있다.

[잔존 점검](#잔존-점검) 절의 정리 대상은 본 세션이 만든 산출물에 한정한다 — 본 세션이 만든 것은 본 세션이 정리, 다른 세션이 만든 것은 그 세션이 정리.
