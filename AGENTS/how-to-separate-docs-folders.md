# _docs/ 하위 폴더 구분 기준

## 폴더별 역할

| 폴더 | 담는 것 | 질문 형태 |
|------|---------|-----------|
| `_ontology/` | 도메인 개념 정의. 의미가 갈릴 수 있는 용어의 구분과 엔티티 관계. | "우리 도메인에 무엇이 존재하는가" |
| `_knowledge/` | 외부 사실, 환경 정보, 벤치마크 분석. | "세상이 어떠한가" |
| `_strategy/` | 우리의 결정과 그 근거. 기각된 대안, 설계 원칙의 이유, 선택의 배경. | "왜 이렇게 하기로 했는가" |
| `_architecture/` | 결정된 구조 자체. 모듈 정의, 의존성 규칙, 빌드/배포 등. | "무엇을 어떻게 만들겠다" |
| `_worklog/` | 작업 상태, 이력, 작업 트리. | (관리용) |

## 판별 규칙

- 같은 단어를 두 사람이 다른 뜻으로 쓸 수 있는가 → `_ontology/`
- 이 내용은 우리 결정과 무관하게 존재하는 외부 사실인가 → `_knowledge/`
- 이 내용을 바꾸면 "왜 그렇게 했나"에 대한 설명만 달라지는가 → `_strategy/`
- 이 내용을 바꾸면 실제 코드/빌드/배포가 달라지는가 → `_architecture/`

## 교차 링크

한 주제가 여러 폴더에 걸칠 수 있다. 이때 중복을 피하려면:

- 개념 정의(용어 구분, 엔티티 관계)는 `_ontology/`에만 둔다.
- 외부 사실(벤치마크, 환경 조건)은 `_knowledge/`에만 둔다.
- 근거(이유/배경)는 `_strategy/`에만 둔다.
- 명세(규칙/구조)는 `_architecture/`에만 둔다.
- 서로 필요한 곳에서 링크한다.
