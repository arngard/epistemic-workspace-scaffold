# Conventions

워크스페이스 공통의 문서 작성 규약과 구조 규약. 사람과 AI 도구 모두 생성, 수정 시 본 규약을 준수한다.

본 문서는 일반 작성 규약과 구조 규약의 SSOT. AGENTS.md "문서 작성 스타일"은 진입자가 첫 화면에서 봐야 할 핵심 항만 유지하고, 그 외 일반 규약은 본 문서에 둔다. 프로젝트 고유의 코드, 빌드 규약은 [_docs/_architecture/conventions.md](../_docs/_architecture/conventions.md)에 둔다 - 그쪽은 인스턴스가 자기 도메인으로 채우는 자리다.

## 일반 문서 작성 스타일

진입자가 첫 화면에서 봐야 할 핵심(평서문, 서식 절제, 마크다운 링크, cf./ref., 약어와 줄임말, front matter, 응답 언어, 추측 금지, 사실/의견 분리)은 [AGENTS.md "문서 작성 스타일"](../AGENTS.md)에서 본다. 본 절은 그 외 일반 규약 본문.

### 1관심사 원칙

하나의 문서는 하나의 관심사를 다룬다. 두 개의 독립된 주제가 한 파일에 있으면 분량과 무관하게 분리한다.

### 분량 보조 지표

한글 기준 2,000자 권장 / 3,000자 상한. 단, "하나의 관심사" 원칙이 본 수치보다 앞선다 - 상한 이내여도 두 주제면 분할하고, 상한을 넘더라도 단일 주제로 응집되면 유지한다.

문서 유형에 따라 분량 상한이 달라진다. 시간 축 문서(날짜 프리픽스가 붙는 히스토리 문서)는 시간순 누적이 본질이므로 상한이 완화된다. 폴더 디스크립터는 폴더 직속 항목 수가 늘수록 자연히 길어질 수 있다. 유형별 임계값은 감사 스크립트가 관리한다 (cf. [tools.md](tools.md) "기계 감사 스크립트").

### 중복 금지

같은 사실, 규칙, 설명을 두 곳 이상에 적지 않는다. 한 사실은 하나의 SSOT 문서에만 두고 다른 문서는 링크로 연결한다.

본 원칙은 SSOT가 외부 시스템(이슈 트래커, 사내 위키, 다른 저장소 등)에 있는 정보에도 적용된다. 외부가 SSOT인 계속 갱신되는 정보는 본문에 옮겨 적지 않고 링크로 가리킨다 - 복제본은 원본의 갱신을 따라가지 못한다. 이 워크스페이스가 더하는 것은 복제가 아니라 종합이다: 여러 출처에 걸친 공통 구조의 추상화, 워크스페이스 내 다른 문서와의 연결, 출처들을 가로질러야 얻어지는 도출. 판단 기준: 원본이 갱신되면 그 서술도 무효가 되는가, 그리고 원본에 없는 내용을 더하지 않는가. 둘 다 그렇다면 복제하지 말고 링크한다.

예외: 출처가 휘발적이어서 사라질 수 있거나, 특정 시점 상태의 보존이 필요하다고 판단되면 내용을 가져온다. 이때는 시간 스냅샷 문서(아래 "문서 시간성")로 두거나, raw 형태면 `_reference/`에 보관한다.

### 이름과 식별자의 안정성

여러 문서에서 참조되는 이름, 용어, 식별자는 시간이 지나면 바뀔 수 있다. 바뀔 수 있는 것을 여러 문서에 박제하면 이후 대량 갱신 부채가 쌓인다. 새 용어를 도입할 때의 검증은 [_docs/_ontology/AGENTS.md](../_docs/_ontology/AGENTS.md) "용어 도입 전 검증" 참조.

계획 순서 식별자(Phase 1, Milestone 2 등)는 [_docs/_worklog/TASK_TREE.md](../_docs/_worklog/TASK_TREE.md)와 프로젝트 개요 문서에만 두고, 나머지 문서에서는 "첫 출시", "후속 확장"처럼 의미 단위 표현으로 참조한다. 순서는 재편될 수 있지만 의미 단위는 덜 변한다.

### 폴더 구조와 디스크립터

문서가 늘면 폴더를 평평하게 두고 중앙 색인에 표를 쌓는 방식은 색인 비대를 부른다. 인간의 폴더 관리처럼 하위 폴더 분화를 기본 구조로 인정하고, 색인 기능을 폴더 단위로 분산한다.

- 폴더 분화: 폴더 직속 항목이 많아지면 하위 폴더로 분류한다. 의미 단위 분류가 기본이고, 동등한 값의 대량 나열(도서 목록 성격)이면 순차 분류(알파벳, 날짜)를 쓴다. 폴더 직속 파일 수 임계는 감사 스크립트가 검출해 분화를 권고한다.
- 폴더 디스크립터: 모든 폴더는 `AGENTS.md`를 가진다. Claude Code(중첩 CLAUDE.md), OpenAI Codex(중첩 AGENTS.md) 등 도구의 계층 자동 인식을 받기 위한 명명이다. 루트 AGENTS.md는 전역 진입점, 폴더 AGENTS.md는 해당 서브트리의 로컬 안내이며, 규칙이 충돌하면 하위가 자기 서브트리에 우선한다. 담는 것: 폴더의 목적, 로컬 규칙(명명 규칙, 시간성 기본값 등), 하위 항목 안내. 폴더를 신설하면 디스크립터를 같은 커밋에 동반한다.
- 규범성에 따른 파일별 안내 의무: 규범 문서(지침, 규약 - 존재를 모르면 위반 행동이 나온다)를 담은 폴더는 디스크립터에 파일별 설명과 "언제 읽는가" 트리거 명시가 의무다. 참조 문서(지식 노드, raw 자료 - 필요 시 검색으로 발견 가능)의 폴더는 폴더 설명과 자기설명적 파일명이 기본이고, 파일별 표는 선택이다.
  - 골격 기본 규범 폴더: `AGENTS/`와 `_docs/_architecture/`. 설계 명세는 구현 시 전체가 함께 참조되어 서로를 지지하는 정보라서(문서 하나하나가 개별적, 확률적으로 가치를 갖는 `_strategy/`와 대비된다) 존재를 모르면 위반 구현이 나오는 규범 문서다. 규범 폴더의 하위 폴더도 모두 규범 폴더로 본다.
  - 그 외 폴더를 규범 폴더로 운영하려면 디스크립터에 파일별 안내 표를 두는 것으로 충분하다. 감사 스크립트는 골격 기본 규범 폴더와, 파일별 안내 표(폴더 직속 문서로의 링크를 담은 표)를 가진 디스크립터를 파일별 안내 완전성 검사 대상으로 삼는다.
- 존재의 SSOT: "어떤 문서가 존재하는가"의 SSOT는 중앙 색인 표가 아니라 파일시스템 자체다. 색인 표의 수동 동기화 부채(등재 누락)가 원리적으로 소멸한다. 처리 대기 등 행동 유발 항목은 해당 폴더 디스크립터 상단에 유지한다. `_reference/`의 디스크립터는 예외로, 인벤토리별 처리 상태(처리 대기, 처리됨, 사용자 의견 필요 등)의 진척 SSOT 역할을 계속 가진다 - 처리 상태는 파일시스템으로 대체할 수 없는 정보다.
- `_implementation/` 예외: 폴더 분화 원칙과 디스크립터 의무는 `_implementation/` 하위 프로젝트 내부(소스 트리)에 적용하지 않는다. 소스 폴더 구조는 해당 코드 생태계의 관례(패키지 구조, 빌드 도구 규약)가 지배하는 영역이다. 본 규약의 소관은 `_implementation/` 직할의 `AGENTS.md`와 그 하위 문서 폴더 `AGENTS/`뿐이다.

### 문서 시간성 (히스토리 문서와 현재적 문서)

문서는 시간성에 따라 두 종류로 나뉘며, 파일명의 날짜 프리픽스 유무가 그 구분을 드러낸다.

- 히스토리 문서(시간 스냅샷 계열): 시간 축을 한눈에 보는 것이 목적이라 누적이 본질이다. 외부에서 받은 자료의 정리, 특정 시점의 상황, 벤치마크 기록, 일자별 리포트 등. 파일명 앞에 `yyyy-MM-dd` 프리픽스를 단다. 서술 슬러그가 붙으면 하이픈으로 잇고(`yyyy-MM-dd-<슬러그>.md`), 날짜만으로 충분하면 `yyyy-MM-dd.md`. 읽을 때 어느 시점의 것인지를 반드시 고려한다.
- 현재적 문서: 현재 상태를 반영하는 문서. 온톨로지 정의, 유효한 전략과 결정, 설계 명세, 규약 등. 파일명에 날짜를 달지 않고, 내용이 바뀌면 즉시 갱신한다.

front matter의 `date created`, `date modified`는 두 종류 모두에 붙는다(ref. [AGENTS.md "문서 작성 스타일"](../AGENTS.md)). 날짜 프리픽스는 그와 별개로, 시간 고정 여부를 파일명에서 즉시 식별하게 하는 표식이다. 현재적 문서의 갱신 이력은 front matter가, 히스토리 문서의 기준 시점은 파일명 프리픽스가 가리킨다.

#### 시간성 혼합 금지

히스토리 문서는 누적이 본질이지만, 현재적 문서는 최신 상태 서술만 담는다. 현재적 문서 본문에 시간순 기록(관측 로그, 발언 인용, 변경 서사)을 누적하지 않는다. 이력은 git과 히스토리 문서가 담당한다.

- 현재적 문서에 시간 축 기록 수요가 생기면 그 부분을 히스토리 문서(동반 로그 파일 또는 기존 시간 축 영역)로 분리하고, 현재적 문서에는 현재 상태 요약과 링크만 남긴다. 분리 방식(동반 로그 파일 vs 시간 축 문서로만 환류)은 워크스페이스 자율이며, 채택한 방식을 해당 폴더 디스크립터에 명시한다.
- 근거 보존 예외: 특이한 규칙, 결정이 미래에 이유를 모른 채 원복될 위험이 예상될 때에 한해, 해당 현재적 문서에 "왜 이래야 하는가"의 근거 요지를 참고 표기로 남긴다. 경위 서사가 아니라 근거 요지만 남긴다.
- tombstone 금지: 현재적 문서를 폐기, 대체할 때 tombstone을 만들지 않는다. 전략, 결정이 재해석으로 대체되면 그 내용은 새 관점의 문서에 재작성하고 구 문서는 삭제한다. `status: 폐기됨` 같은 태그를 달아 잔존시키는 운영을 하지 않는다. 과거 판의 보관소는 git 이력이다.

#### 표준 문서 유형의 시간성

표준 문서 유형의 시간성을 아래로 확정한다. 개별 판단에 맡기지 않는다.

- 현재적 문서: AGENTS.md와 `AGENTS/` 산하 에이전트 지침 전부, `_docs/_ontology/` 정의, `_docs/_knowledge/` 노드(현재 상태 서술), `_docs/_strategy/`의 유효한 전략과 결정 문서, `_docs/_architecture/` 명세, 모든 폴더 디스크립터(현재 인벤토리), `_docs/_worklog/TASK_TREE.md`(현재와 가까운 전후의 작업 상태를 표현하며 이력 보관소가 아니다), `_docs/_worklog/STATUS.md`(상태 레지스트리).
- 히스토리 문서: 날짜 프리픽스가 붙는 모든 문서(데일리, 먼슬리 리포트, 시점 분석, 벤치마크 기록), 현재적 노드에서 분리된 동반 로그 파일, `_reference/` raw 자료(immutable).

#### 날짜 프리픽스의 의미와 종결 신호

날짜 프리픽스는 사안의 시작 시점 앵커이지 완결 표식이 아니다. 히스토리 문서는 해당 사안이 진행 중인 동안 계속 갱신될 수 있다(갱신 여부는 front matter `date modified`가 드러낸다). 사안이 종결된 뒤에는 원 문서를 수정하지 않는다 - 종결 후 동결 규칙이다.

종결을 에이전트가 판별 가능한 신호로 만든다.

- front matter 선택 필드 `date closed`를 도입한다. 히스토리 문서 전용이다 - 현재적 문서에는 종결 개념이 없다(유효하면 최신화, 대체되면 삭제). 값이 있으면 그 시점부로 동결이며, 본문을 수정하지 않고 갱신 수요는 새 날짜의 새 문서로 다룬다. 동결 문서에 허용되는 편집은 후속 문서 링크 추가와 워크스페이스 관리상 필요한 정합 유지 수정(링크 경로 정정, 규약 변경에 따른 front matter 갱신 등)뿐이다. 본문의 의미 내용은 변경하지 않는다. 값이 없으면 사안이 진행 중이며 갱신 가능하다.
- 영역 단위 기본값: 시간 축 영역의 폴더 디스크립터는 소속 문서의 기본 수명을 선언할 수 있다(예: "데일리 리포트는 발행 즉시 종결"). 개별 문서의 `date closed`가 영역 기본값에 우선한다.
- 에이전트 편집 규칙: 날짜 프리픽스 문서를 수정하기 전에 종결 여부를 개별 필드, 영역 기본값 순으로 확인한다.

### 코드 블록 절제

문서 내 예시 코드 블록은 서술만으로 전달하기 어려운 구조에만 사용한다. 정책이나 규칙은 서술로 충분하면 코드 블록을 두지 않는다. (본 규칙은 문서 안의 코드 블록에 한정되며, `_implementation/` 하위의 실제 구현 코드에는 적용되지 않는다.)

### 표준 표현 우선

비유적 표현, 관용구, 글쓴이만 아는 함축은 창작적 글에서는 허용되지만 기술 문서, 지침서, 코드에서는 쓰지 않는다. 이런 글은 정확히 이해돼야 하는데, 문자 그대로가 아닌 표현은 독자가 뜻을 추론해야 해서 오독 위험이 크다. 널리 통용되는 표준 표현을 직설적으로 쓴다.

- 판단 기준: 이 표현을 처음 보는 독자(사람이든 AI든)가 정확히 이해하는가.
- 반복을 피하려고 표현을 억지로 다변화하거나 뜻이 흐릿한 표현을 끌어들이지 않는다.
- 뜻에 맞는 정확한 동사를 고른다. 다만 의미가 불확실한 비유적 표현을 남발하지 않는다.

### 출처와 주체 표기의 범위

사실과 의견을 구분하고 의견에 주체를 명시하는 원칙은 [AGENTS.md "문서 작성 스타일"](../AGENTS.md) 참조. 본 절은 그 표기를 어디까지 하는가를 정한다.

- 표기가 필요한 것: 의견, 주장, 분석, 판단의 주체. 검증이 필요하거나 논쟁적인 사실, 특정 외부 출처에서 온 사실의 출처.
- 표기가 필요 없는 것: 프로젝트 참여자 누구나 같게 받아들이는 공유 전제. 이런 사실에까지 출처를 다는 것은 잡음이며, 정작 출처가 중요한 진술을 가린다.
- 판단 기준: 이것이 누군가의 의견이나 주장이라서 다른 사람은 달리 볼 수 있는가. 그렇다면 표기하고, 공유된 전제라면 표기하지 않는다.
- 결격 대응: 주체나 출처가 누락된 의견, 정보를 발견하면 추측으로 채우지 않고 작성자에게 보완을 요청한다. 임의로 추정해 적지 않는다.

### 지침과 사실 서술의 자체 완결성

현재적 문서(지침, 규약, 정의, 설계 명세, 사실 정리)는 현재 유효한 내용만 담백하게 담는다. 그 내용이 정립된 경위, 논의 히스토리, 원자료를 추적하는 서술을 본문에 함께 두지 않는다 - 독자의 주의를 분산시키고 문서의 완결성과 가독성을 해친다. 근거나 참고가 필요하면 `ref.`, `cf.` 링크로 간결히 가리키는 데 그친다. 원자료나 히스토리를 읽어야만 본문이 이해되는 구성으로 만들지 않는다.

### 고정 규격 문서 태그 중복 회피

파일명 자체가 문서의 시스템적 역할을 지시하는 고정 규격 문서(README, AGENTS, STATUS, TASK_TREE 등)는, 그 파일명과 같은 의미의 단어가 태그에 중복되는 경우 해당 태그를 생략한다. 파일명이 콘텐츠 주제를 나타내는 일반 문서는 제목이 바뀔 수 있으므로 생략하지 않는다.

### 도구 전용 설정 파일 예외

도구 전용 설정 파일(`.claude/`, `.gemini/`, `.github/` 등 특정 AI 도구가 고정된 스키마로 읽는 파일)은 해당 도구의 스키마를 우선한다. AGENTS.md "문서 작성 스타일"의 공통 필수 필드(`date created`, `date modified`, `tags`)를 본 파일들에 강제하지 않는다.

### 문장 부호 표기 (ASCII 우선)

문장 부호는 ASCII(American Standard Code for Information Interchange) 코드를 사용한 표기를 우선한다. 비ASCII 부호 중 ASCII로 자연 대체 가능한 것은 ASCII로 작성한다. 근거: 검색, grep, diff, 외부 도구 호환성, 키보드 직접 입력 가능성.

대상 문자와 치환:

- 가운뎃점(`·`, U+00B7): 항목 구분, 병렬 나열에는 쉼표(`, `)를 쓴다. 두 항목의 양자택일/대안에는 슬래시(`/`)가 허용된다.
- em dash(`—`, U+2014): 부연 설명에는 공백 하이픈 공백(` - `)을 쓰거나 문장을 분리한다(`. `). 정의나 예시를 끌어낼 때는 콜론(`: `). 범위 표시는 물결표(`~`)나 "에서 ~까지" 서술을 쓴다.
- 화살표(`→`, U+2192 / `←`, U+2190): 흐름, 단계, 상태 변화에는 `->`(또는 `<-`)를 쓴다. 단순 참조 포인터는 `ref.`나 서술로 푼다.
- 말줄임표(`…`, U+2026): 마침표 세 개(`...`)나 "등"으로 쓴다.
- 둥근 따옴표(`“` `”` `‘` `’`, U+201C/201D/2018/2019): ASCII 따옴표(`"`, `'`)를 쓴다.

- 권장: "프런트엔드, 백엔드 동시 배포", "개발, 스테이징, 프로덕션 환경", "읽기, 쓰기, 실행 권한"
- 금지: "프런트엔드·백엔드 동시 배포", "개발·스테이징·프로덕션 환경", "읽기·쓰기·실행 권한"

보존 예외(ASCII로 바꾸지 않는다):

- 규칙이나 설명에서 그 문자 자체를 가리키는 경우(본 절의 백틱 안 예시처럼).
- 사용자 발언이나 외부 텍스트를 그대로 옮긴 직접 인용.
- 백틱 안의 코드, 명령, 파일명, 식별자.

본 규칙은 신규 작성과 기존 문서 모두에 적용한다.

## 폴더, 파일 이동 시 연쇄 갱신

폴더, 파일 이동 또는 리네임은 연쇄 갱신을 요구한다. 구조 변경 한 번이 다수 문서의 링크, 태그, 디스크립터 안내, 상대경로에 수작업 파급을 유발한다. 반쪽만 정정된 상태로 남으면 이후 수습 비용이 급격히 커진다. 예방이 사후 수습보다 항상 싸다.

같은 커밋 안에서 다음을 일괄로 수행한다:

1. 이동된 파일을 참조하는 모든 문서의 상대경로 갱신 (`grep -rn "구경로" . --include="*.md"`로 워크스페이스 전수 확인 - `_docs/`, `AGENTS/` 외 `.claude/`, `_implementation/` 등 다른 위치의 마크다운도 포함).
2. front matter `tags`의 첫 항목이 새 소속 카테고리와 일치하는지 확인.
3. 관련 폴더 디스크립터(AGENTS.md)의 안내 갱신 (규범 폴더의 파일별 안내, `_reference/` 처리 상태 표 등 - 기존 항목 수정 또는 삭제, 새 위치에 반영).
4. 자매 폴더(`_strategy`/`_knowledge`/`_ontology`/`_architecture` 간) 일괄 작업이라면 한쪽만 반영하고 다른 쪽이 누락되지 않았는지 교차 확인.
5. 개편 규모가 크면 종료 전 epistemic-auditor를 호출해 사후 문서 정합성 감사를 받는다. 체크리스트(예방)와 감사(치료)는 역할이 다르므로 둘 다 수행한다. cf. [agent-roles.md](agent-roles.md).
