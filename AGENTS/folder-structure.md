# 폴더 구조 규약

폴더를 어떻게 조직하고, 구조 변경 시 무엇을 갱신하는가의 규약. 문서 단위와 SSOT 원칙은 [document-units.md](document-units.md) 참조.

## 폴더 구조와 디스크립터

문서가 늘면 폴더를 평평하게 두고 중앙 색인에 표를 쌓는 방식은 색인 비대를 부른다. 인간의 폴더 관리처럼 하위 폴더 분화를 기본 구조로 인정하고, 색인 기능을 폴더 단위로 분산한다.

- 폴더 분화: 폴더 직속 항목이 많아지면 하위 폴더로 분류한다. 의미 단위 분류가 기본이고, 동등한 값의 대량 나열(도서 목록 성격)이면 순차 분류(알파벳, 날짜)를 쓴다. 폴더 직속 파일 수 임계는 감사 스크립트가 검출해 분화를 권고한다.
- 폴더 디스크립터: 모든 폴더는 `AGENTS.md`를 가진다. Claude Code(중첩 CLAUDE.md), OpenAI Codex(중첩 AGENTS.md) 등 도구의 계층 자동 인식을 받기 위한 명명이다. 루트 AGENTS.md는 전역 진입점, 폴더 AGENTS.md는 해당 서브트리의 로컬 안내이며, 규칙이 충돌하면 하위가 자기 서브트리에 우선한다. 담는 것: 폴더의 목적, 로컬 규칙(명명 규칙, 시간성 기본값 등), 하위 항목 안내. 폴더를 신설하면 디스크립터를 같은 커밋에 동반한다.
- 규범성에 따른 파일별 안내 의무: 규범 문서(지침, 규약 - 존재를 모르면 위반 행동이 나온다)를 담은 폴더는 디스크립터에 파일별 설명과 "언제 읽는가" 트리거 명시가 의무다. 참조 문서(지식 노드, raw 자료 - 필요 시 검색으로 발견 가능)의 폴더는 폴더 설명과 자기설명적 파일명이 기본이고, 파일별 표는 선택이다.
  - 골격 기본 규범 폴더: `AGENTS/`와 `_docs/_architecture/`. 설계 명세는 구현 시 전체가 함께 참조되어 서로를 지지하는 정보라서(문서 하나하나가 개별적, 확률적으로 가치를 갖는 `_strategy/`와 대비된다) 존재를 모르면 위반 구현이 나오는 규범 문서다. 규범 폴더의 하위 폴더도 모두 규범 폴더로 본다.
  - 그 외 폴더를 규범 폴더로 운영하려면 디스크립터에 파일별 안내 표를 두는 것으로 충분하다. 감사 스크립트는 골격 기본 규범 폴더와, 파일별 안내 표(폴더 직속 문서로의 링크를 담은 표)를 가진 디스크립터를 파일별 안내 완전성 검사 대상으로 삼는다.
- 존재의 SSOT: "어떤 문서가 존재하는가"의 SSOT는 중앙 색인 표가 아니라 파일시스템 자체다. 색인 표의 수동 동기화 부채(등재 누락)가 원리적으로 소멸한다. 처리 대기 등 행동 유발 항목은 해당 폴더 디스크립터 상단에 유지한다. `_reference/`의 디스크립터는 예외로, 인벤토리별 처리 상태(처리 대기, 처리됨, 사용자 의견 필요 등)의 진척 SSOT 역할을 계속 가진다 - 처리 상태는 파일시스템으로 대체할 수 없는 정보다.
- `_implementation/` 예외: 폴더 분화 원칙과 디스크립터 의무는 `_implementation/` 하위 프로젝트 내부(소스 트리)에 적용하지 않는다. 소스 폴더 구조는 해당 코드 생태계의 관례(패키지 구조, 빌드 도구 규약)가 지배하는 영역이다. 본 규약의 소관은 `_implementation/` 직할의 `AGENTS.md`와 그 하위 문서 폴더 `AGENTS/`뿐이다.
- 실체 0건 골격 폴더: 스캐폴드가 출고하는 표준 폴더(`_docs/` 4범주, `_worklog/`, `_reference/`, `_implementation/` 등)는 디스크립터만 있고 실체(직속 문서)가 0건이어도 정상이다 - 첫 실체를 기다리는 상태다. 새 폴더를 신설할 때 첫 실체 1건을 같은 단위 작업에 포함해야 하는 규칙(cf. [AGENTS.md](../AGENTS.md))은 새로 만드는 폴더에 적용되며, 골격 표준 폴더의 출고 상태에는 적용되지 않는다. 감사 스크립트가 이 표준 폴더 목록을 실체 0건 검출의 예외로 관리한다.

## 폴더, 파일 이동 시 연쇄 갱신

폴더, 파일 이동 또는 리네임은 연쇄 갱신을 요구한다. 구조 변경 한 번이 다수 문서의 링크, 태그, 디스크립터 안내, 상대경로에 수작업 파급을 유발한다. 반쪽만 정정된 상태로 남으면 이후 수습 비용이 급격히 커진다. 예방이 사후 수습보다 항상 싸다.

같은 커밋 안에서 다음을 일괄로 수행한다:

1. 이동된 파일을 참조하는 모든 문서의 상대경로 갱신 (`grep -rn "구경로" . --include="*.md"`로 워크스페이스 전수 확인 - `_docs/`, `AGENTS/` 외 `.claude/`, `_implementation/` 등 다른 위치의 마크다운도 포함).
2. front matter `tags`의 첫 항목이 새 소속 카테고리와 일치하는지 확인.
3. 관련 폴더 디스크립터(AGENTS.md)의 안내 갱신 (규범 폴더의 파일별 안내, `_reference/` 처리 상태 표 등 - 기존 항목 수정 또는 삭제, 새 위치에 반영).
4. 자매 폴더(`_strategy`/`_knowledge`/`_ontology`/`_architecture` 간) 일괄 작업이라면 한쪽만 반영하고 다른 쪽이 누락되지 않았는지 교차 확인.
5. 개편 규모가 크면 종료 전 epistemic-auditor를 호출해 사후 문서 정합성 감사를 받는다. 체크리스트(예방)와 감사(치료)는 역할이 다르므로 둘 다 수행한다. cf. [agent-roles.md](agent-roles.md).
